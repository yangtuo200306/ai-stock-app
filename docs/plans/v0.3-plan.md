# v0.3 AI 问股可用版计划

## 1. 版本定位

v0.3 的目标是让 AI Stock App 从“基础规则分析”升级为“更可用的 AI 问股助手”。

核心原则：

- 借鉴原参考项目思想，但不照搬复杂架构。
- 先提升行情数据稳定性，再接入大模型。
- 大模型基于真实行情、技术指标和规则分析生成回答。
- 每个阶段都要能独立验收。
- 继续保持学习项目的小步闭环，不扩成完整交易软件。

v0.3 的整体方向：

```text
数据更稳
  ↓
问股更像 AI
  ↓
项目真正更可用
```

## 2. 当前基础

v0.3 基于 v0.2 已完成能力继续推进。

当前项目已有：

1. v0.1 MVP 闭环：
   - 自选股管理。
   - 生成分析任务。
   - 查询任务状态。
   - 生成报告。
   - 查看报告详情。

2. v0.2 基础分析增强：
   - MA5 / MA10 / MA20。
   - 乖离率。
   - 均线趋势。
   - 动态评分。
   - 动态建议。
   - 风险提示。
   - 历史报告列表增强。
   - 问股最小版。
   - 我的页基础整理。

3. 当前行情能力：
   - 主数据源：`efinance`。
   - 已预留 fallback 函数。
   - 但 fallback 尚未真正接入备用源。

4. 当前问股能力：
   - 支持输入股票代码。
   - 后端返回基于行情和指标的规则型基础分析。
   - 尚未调用大模型。

## 3. v0.3 总体目标

v0.3 总体目标：

1. `efinance` 失败时，自动尝试新浪备用源。
2. 问股功能接入用户已有大模型。
3. 用户输入股票问题后，能获得更自然的 AI 分析回答。
4. AI 回答必须基于真实行情、技术指标和规则分析上下文。
5. 大模型失败时，不让功能完全不可用，而是回退到当前规则分析。

最终体验目标：

```text
用户输入：600519 怎么看？
  ↓
后端提取股票代码
  ↓
获取实时行情和历史行情
  ↓
计算 MA 指标和基础评分
  ↓
构造大模型上下文
  ↓
返回 AI 风格分析回答
```

## 4. 功能范围

### 必做范围

1. 双源 fallback
   - 保留 `efinance` 为主数据源。
   - 新增新浪直连作为备用数据源。
   - 覆盖实时行情 fallback。
   - 覆盖历史行情 fallback。
   - 成功时返回实际 `source`。
   - 两个源都失败时，返回聚合错误。

2. 新浪实时行情直连
   - 使用新浪实时行情接口。
   - 支持 A 股 6 位代码转新浪 symbol。
   - 解析股票名称、当前价、昨收价。
   - 计算涨跌幅。
   - 转换成现有 `StockQuote`。

3. 新浪历史 K 线直连
   - 使用新浪历史 K 线接口。
   - 获取最近至少 20 条日线数据。
   - 解析日期、开盘、收盘、最高、最低、成交量。
   - 转换成现有 `StockDailyPrice`。

4. AI 问股最小可用版
   - 问股支持输入包含 6 位股票代码的问题。
   - 后端从问题中提取 6 位股票代码。
   - 后端复用行情、指标和规则分析能力。
   - 后端构造大模型上下文。
   - 后端调用用户已有大模型。
   - 返回 AI 风格回答。
   - 大模型失败时回退规则回答。

5. 问股页面增强
   - 保留当前问股页结构。
   - 展示用户问题。
   - 展示 AI 回答。
   - 展示基础行情和指标依据。
   - 展示风险提示。

### 选做范围

1. 轻量错误提示增强
   - 在不重构全局错误系统的前提下，让 fallback 错误更容易理解。

2. Prompt 版本标记
   - 可在后端记录当前 prompt 版本，方便后续调试。

3. 大模型响应来源标记
   - 可返回 `answer_type = ai` 或 `answer_type = rule_fallback`。

## 5. 不做范围

v0.3 暂不做：

- 不引入 AKShare。
- 不引入 Tushare。
- 不引入 Baostock。
- 不做完整 `DataFetcherManager`。
- 不做多源优先级配置。
- 不做缓存系统扩展。
- 不做熔断器。
- 不做重试队列。
- 不做港股 / 美股等多市场支持。
- 不做股票中文名复杂识别。
- 不做多轮聊天。
- 不做流式输出。
- 不保存聊天历史。
- 不做会话管理。
- 不做登录注册。
- 不做用户隔离。
- 不做推送提醒。
- 不做回测系统。
- 不改数据库结构，除非实施前重新确认。

## 7. 数据库影响

v0.3 默认不改数据库表结构。

原因：

- 阶段 1 fallback 不需要保存新字段。
- 阶段 2 AI 问股最小版不保存聊天历史。
- 当前问股仍然是“即时回答”，不是“会话记录”。

如果后续要保存问股历史，再单独设计表结构，例如：

```text
ask_messages
ask_sessions
```

但不属于 v0.3 当前范围。

## 8. 推荐实施阶段

### 阶段 1：efinance + 新浪直连 fallback

目标：

- 主源 `efinance` 失败后，自动尝试新浪直连备用源。
- 实时行情和历史行情都支持 fallback。
- 报告生成和问股功能在主源失败时仍有机会继续完成。

涉及：

- `backend/requirements.txt`
- `backend/app/services/market_data.py`

实现要点：

1. 新增 `requests` 依赖。
2. 新增 `FALLBACK_MARKET_SOURCE = "sina"`。
3. 新增新浪实时接口常量。
4. 新增新浪历史 K 线接口常量。
5. 新增浏览器请求头常量。
6. 新增 `_to_sina_symbol(stock_code)`。
7. 实现 `_get_stock_quote_from_fallback(...)`。
8. 实现 `_get_stock_history_from_fallback(...)`。
9. 修改 `get_stock_quote(...)`，主源失败后尝试 fallback。
10. 修改 `get_stock_history(...)`，主源失败后尝试 fallback。
11. 两个源都失败时返回聚合错误。

实时行情映射：

```text
fields[0]  → name
fields[2]  → 昨收
fields[3]  → 当前价
change_pct = (当前价 - 昨收) / 昨收 * 100
source = sina
```

历史行情映射：

```text
day/date → date
open     → open
close    → close
high     → high
low      → low
volume   → volume
```

验收：

- 无效代码 `abc` 仍返回“目前仅支持 6 位 A 股代码”。
- `GET /api/market/quote/600519` 能返回行情或清晰聚合错误。
- efinance 正常时，原行为不变。
- efinance 失败时，后端尝试新浪 fallback。
- 新浪成功时，返回 `source = sina`。
- `POST /api/analysis` 不被破坏。
- `POST /api/ask` 不被破坏。

### 阶段 2：AI 问股最小可用版

目标：

- 问股从规则回答升级为大模型回答。
- 用户可以输入包含股票代码的问题，例如：

```text
600519 怎么看？
000001 风险大吗？
帮我分析一下 300750
```

涉及：

- `backend/app/api/ask.py`
- `backend/app/services/llm_client.py`
- `backend/app/services/report_builder.py`（可选）
- `mobile-app/src/screens/AskScreen.tsx`
- `mobile-app/src/types/index.ts`
- `backend/requirements.txt`（视大模型调用方式决定）

实现要点：

1. 后端从用户问题中提取 6 位股票代码。
2. 若没有 6 位股票代码，返回明确错误。
3. 复用 `get_stock_quote(...)`。
4. 复用 `get_stock_history(...)`。
5. 复用 `build_technical_indicators(...)`。
6. 复用 `build_analysis_report(...)`。
7. 构造大模型上下文。
8. 调用用户已有大模型。
9. 大模型成功时返回 AI 回答。
10. 大模型失败时回退当前规则回答。
11. 前端展示 AI 回答、基础行情、指标和风险提示。

Prompt 原则：

```text
角色：股票分析助手。
依据：只能基于后端提供的行情、指标、评分和风险提示。
禁止：编造实时行情；禁止给出确定性买卖指令。
要求：明确“仅供学习参考，不构成投资建议”。
风格：简洁、清楚、适合新手理解。
```

返回字段建议：

```text
stock_code
stock_name
price
change_pct
trend
action
score
question
answer
answer_type
risks
indicators
model
```

验收：

- 输入 `600519 怎么看` 能返回 AI 风格回答。
- 回答内容包含真实行情和指标依据。
- 回答中包含风险提示。
- 大模型失败时仍能显示规则分析。
- API Key 不暴露到前端。
- 不影响报告生成、历史报告、自选股和我的页。

## 9. 大模型接入约定

大模型由用户提供，实施阶段再确认具体参数。

建议配置项：

```text
LLM_API_KEY=
LLM_BASE_URL=
LLM_MODEL=
```

要求：

- API Key 只放后端环境变量。
- 前端不保存、不展示、不传输 API Key。
- 如果是 OpenAI 兼容接口，优先使用兼容格式。
- 如果不是 OpenAI 兼容接口，阶段 2 实施前重新确认调用格式。

## 10. 原项目可借鉴点

### 数据源 fallback

参考原项目 `data_provider` 思想：

- 不同数据源返回不同格式。
- 系统内部要统一成同一结构。
- 主源失败后尝试备用源。
- 错误信息应保留每个源的失败原因。

v0.3 只借鉴思想，不照搬完整 `DataFetcherManager`。

### AI 问股

参考原项目 Agent / Chat 思想：

- 使用 `system` + `user` 的 messages 结构。
- Prompt 包含角色、数据、规则、风险提示。
- AI 回答必须基于上下文。
- 风险提示要明确“不构成投资建议”。

v0.3 不照搬：

- LiteLLM 多模型适配。
- 多 Agent 编排。
- 工具调用。
- ReAct 循环。
- 流式输出。
- 会话管理。

## 11. 学习重点

v0.3 主要学习：

1. 多数据源 fallback 的基本思想。
2. 如何把不同数据源转换成统一结构。
3. 如何设计清晰的聚合错误。
4. 为什么业务层不应该关心数据源细节。
5. 大模型为什么需要真实行情上下文。
6. 如何构造最小 Prompt。
7. 如何保护 API Key。
8. 如何设计 AI 失败时的规则回退。
9. 为什么不一开始做完整 Chat 系统。

## 12. 风险点

1. 新浪接口也是非官方接口
   - 可能字段变化。
   - 可能临时不可用。
   - 可能被反爬。
   - 当前只作为低频学习 fallback。

2. 新浪历史 K 线接口不稳定
   - 返回字段可能与预期不同。
   - 复权口径不明确。
   - 如果不稳定，后续再考虑腾讯或其他历史源。

3. 新增 `requests` 依赖
   - 环境需要重新安装依赖。
   - 需要更新 `backend/requirements.txt`。

4. 大模型 API 失败
   - 可能超时。
   - 可能鉴权失败。
   - 可能额度不足。
   - 需要规则回答兜底。

5. AI 回答可能过度像投资建议
   - Prompt 必须限制。
   - 回答必须包含“仅供学习参考”。
   - 不使用“必买 / 必卖 / 保证收益”等表达。

6. 问股响应结构可能继续变复杂
   - v0.3 先保持最小扩展。
   - 后续若做多轮聊天，再重构响应结构。

## 13. v0.3 完成标准

v0.3 完成时应满足：

1. efinance 失败时，系统会尝试新浪 fallback。
2. 报告和问股不再完全依赖单一行情源。
3. 问股可以调用大模型生成自然语言分析。
4. 大模型回答基于真实行情和技术指标。
5. 大模型失败时有规则回答兜底。
6. 前端问股页能展示 AI 回答和分析依据。
7. 文档、学习记录、审查记录和版本总结完整。
