## 1. 项目开发模式

### 工作流

```
1. 制定大计划（overall-roadmap.md）
   → 分若干个阶段，每个阶段有明确目标
   
2. 制定阶段计划（vX.Y-plan.md）
   → 根据大计划拆解当前阶段的具体任务
   
3. 执行阶段
   → 按计划逐步实施
   → 同步更新学习笔记（learning-notes.md）
   
4. 记录问题
   → 代码问题/隐患 → code-review-notes.md
   → 多次出现的高频问题 → common-issues.md
   
5. 阶段完成
   → 生成版本总结（archive/releases/）
   → 更新学习笔记
   
6. 大计划全部完成
   → 生成学习总结（learning-summary.md）
```

### 文档管理原则

- **活跃文档**：当前还在用的文档，放在 `docs/` 根目录或子目录
- **归档文档**：已完成阶段的所有记录（计划、总结、学习笔记），移入 `archive/`
- **参考文档**：开发时频繁查阅的结构/数据流/架构说明，放在 `guide/`
- **经验记录**：个人积累的重要经验，放在根目录 `经验记录.md`，独立醒目

### 协作原则

- 涉及代码修改前先说明目标、涉及文件和学习点
- 大改动前先规划再执行
- 优先做最小可运行版本
- 不擅自提交和推送
- 文档创建/修改前先确认

## 2. 部署经验

### 2.1 服务器系统版本决定一切

- **Ubuntu 20.04** 默认只有 Python 3.8/3.9，很多新库不兼容
- **Ubuntu 22.04+** 自带 Python 3.10+，兼容性更好
- 选服务器系统时，尽量选较新的 LTS 版本，能省去大量兼容性问题

### 2.2 依赖版本策略

| 场景 | 做法 |
|------|------|
| 个人项目（环境不同） | 用 `>=` 最低版本，灵活兼容 |
| 企业生产（环境一致） | 用 `==` 锁定 + lock 文件 |
| 服务器 Python 版本较旧 | 用 `>=` 让 pip 自动选兼容版本 |

### 2.3 Python 3.9 已知兼容问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `str \| None` 语法报错 | Python 3.10+ 语法 | 安装 `eval-type-backport` |
| fastapi 新版装不了 | 0.130+ 需要 3.10+ | pip 自动选 0.128.8 |
| uvicorn 新版装不了 | 0.40+ 需要 3.10+ | pip 自动选 0.39.0 |
| pandas 编译慢 | 没有 3.9 的预编译 whl | 装 `pandas<2.0`（有预编译包） |
| numpy 不兼容 | numpy 2.x 不兼容 pandas 1.x | 装 `numpy<2` |

### 2.4 部署流程要点

1. **代码上传**：仓库设为公开，HTTPS 克隆无需认证
2. **虚拟环境**：用 `python3.x -m venv` 创建，避免污染系统 Python
3. **systemd 服务**：`Restart=always` 确保崩溃自动重启
4. **端口冲突**：`systemctl stop` 后再 `start`，避免 `address already in use`
5. **安全组**：阿里云控制台开放 8000 端口，手机 App 才能访问

## 3. 项目规则编写经验

### 3.1 核心原理：AI 的注意力是稀缺资源

项目规则文件不是给人类看的说明书，而是给 AI 写的程序。AI 的注意力有限：

| 文件长度 | AI 表现 |
|---------|--------|
| **<100 行** | 能记住大部分规则，执行时主动遵守 |
| **100-200 行** | 记住开头和结尾，中间容易忘 |
| **>300 行** | 只记得最后读到的几条，前面的被冲淡 |

所以规则文件要短、要分优先级、要把最重要的放最前面。

### 3.2 三段式结构

行业最佳实践是把规则分成三段：

```
第一段：硬规则（AI 必须遵守，~10 条）
  → 放最前面，AI 一定能记住
  → 用"必须/禁止"，不用"建议"
  → 如：改代码前先查 docs/guide/

第二段：项目事实（AI 需要知道的上下文）
  → 放中间，不要求记住，需要时主动查
  → 如：技术栈、目录说明、文档索引

第三段：用户画像（AI 可以参考的风格指南）
  → 放最后，不强制
  → 如：学习目标、沟通偏好、学习方式
```

### 3.3 具体原则

1. **硬规则不超过 10 条** — 超过 10 条 AI 会在执行时"忘记"某些规则
2. **用"必须/禁止"而不是"建议"** — "改代码前必须先查 guide"比"建议先查文档"有效
3. **删掉重复内容** — 目录结构、版本历史、技术栈如果在其他文档已有，规则文件里不要重复写
4. **不混用户规则和项目规则** — 用户规则（沟通偏好、学习方式）可以放在项目规则第三段，但不要和硬规则混在一起
5. **保留"下次对话启动方式"** — 新对话时 AI 会按这个顺序读文档，确保不会漏

### 3.4 我的实践

v1.8 完成后，项目规则从 **340 行 → 121 行**，砍了 64%：

| | 改前 | 改后 |
|---|---|---|
| 行数 | 340 行 | 121 行 |
| 结构 | 13 节平铺 | 四段式 |
| 硬规则 | 混在 13 节里 | 前 6 条集中 |
| 项目事实 | 散落各处 | 第二节集中 |
| 用户画像 | 在多个章节 | 第三节集中 |

效果：AI 能记住硬规则，执行时主动查文档，不再"失忆"。
