# v1.9.4 轻量 Agent（Function Calling）实施总结

> 在现有架构上新增 Agent 端点，让 LLM 通过 Function Calling 自主决定调用哪些数据工具，实现"按需取数、智能编排"。

---

## 一、设计原则

- **不动现有路径**：`POST /api/ask` 和 `POST /api/ask/stream` 保持稳定
- **新增 Agent 端点**：`POST /api/ask/agent/stream`（仅流式）
- **两种模式可切换**：右上角按钮切换 Agent / 标准模式，默认标准模式
- **三层降级**：Agent 失败 → 回退传统流式 → 非流式 → 规则回答

---

## 二、新增文件

### 1. `backend/app/services/tool_registry.py` — 工具注册表

4 个工具定义（`search_stock` 已排除），每个包含 OpenAI 格式 function schema + handler：

| 工具名 | 功能 | handler |
|--------|------|---------|
| `get_stock_quote` | 获取个股实时行情 | `market_data.get_stock_quote()` |
| `get_technical_indicators` | 获取技术指标（均线/RSI/成交量） | `technical_indicators.build_technical_indicators()` |
| `get_stock_news` | 获取个股相关新闻 | `news_fetcher.fetch_news()` |
| `get_analysis_report` | 获取综合分析报告（评分+趋势+建议） | `report_builder.build_analysis_report()` |

核心数据结构：

```python
class ToolDefinition:
    name: str
    description: str
    parameters: dict  # JSON Schema
    handler: Callable

class ToolRegistry:
    _tools: dict[str, ToolDefinition]
    def register(self, tool: ToolDefinition)
    def get_openai_tools(self) -> list[dict]
    def execute(self, name: str, **kwargs) -> str
```

### 2. `backend/app/services/tool_factory.py` — 工具工厂

注册 4 个工具的 handler 函数，每个 handler 调用现有 Service 并返回结构化 dict。

### 3. `backend/app/services/agent_loop.py` — Agent 执行循环

核心函数 `run_agent_loop()`：

```
输入: system_prompt, messages, tools, max_steps=5
输出: Generator[yield SSE events]

循环:
  1. yield {"type": "thinking", "message": "AI 正在分析..."}
  2. 调用 LLM (带 tool definitions)
  3. 如果 LLM 返回 tool_calls:
     for each tool_call:
       a. yield {"type": "tool_start", "tool": name, "display_name": "..."}
       b. 执行工具 (tool_registry.execute)
       c. yield {"type": "tool_done", "tool": name, "success": bool, "duration": float}
       d. 将 tool 结果追加到 messages
  4. 如果 LLM 返回文本（无 tool_calls）:
     yield {"type": "text", "content": chunk}  (流式逐块)
     break
  5. yield {"type": "done", "success": true, "session_id": "xxx", "full_answer": "..."}
```

保护机制：
- `max_steps=5` 防止死循环
- 单步超时 30s，整体超时 120s
- LLM 调用失败 → yield done {success: false} → 前端降级

```typescript
export interface AgentEvent {

  message?: string;

  display_name?: string;
  success?: boolean;
  duration?: number;

}
```

## 四、修复清单（8 个问题）

| # | 问题 | 根因 | 修复 |
|---|------|------|------|
| 1 | `get_stock_quote` 报错 | `StockQuote` 没有 `volume`/`amount` 字段 | 替换为 `turnover_rate`/`amplitude` |
| 2 | 不是真流式 | LLM 文本累加后一次性 yield | 改为逐 chunk 透传前端 |
| 3 | 记录不保存 | 字符串匹配 `"type":"text"` 和 `json.dumps` 输出 `"type": "text"` 格式不一致 | 改为 `json.loads()` 正确解析 |
| 4 | 多一轮 `search_stock` | tools 列表包含 `search_stock` | 从 tools 排除 |
| 5 | 思考过程步骤不合理 | 每轮循环都发 thinking 事件 | 只发一次 |
| 6 | 回答格式不一致 | Agent system prompt 没结构化要求 | 加 `【结论】【关键数据】【详细分析】【风险提示】` |
| 7 | 新闻没有链接 | 新闻数据没传前端 | system prompt 要求 LLM 输出 `[标题](url)` Markdown 链接 |
| 8 | 内容超出卡片区域 | 没设置溢出样式 | 加 `overflow: hidden` + `flexWrap: wrap` |

## 五、影响面分析

| 维度 | 分析 |
|------|------|
| **数据链路** | Agent 端点内部调用现有 Service，数据流向不变。最终 answer 存入 `ask_messages` 表，字段不变。 |
| **数据库** | 无需新建表、无需加字段、无需迁移。 |
| **前后端类型** | 前端新增 `AgentEvent` / `ThinkingStep` 类型。后端 `AskResponse` 不变。 |
| **错误处理** | 三层降级：Agent 循环失败 → 回退传统流式 → 规则回答。Agent 内部有 max_steps / 超时保护。 |
| **配置** | 无需新增环境变量。 |
| **路由注册** | `ask.py` 的 router 已在 `main.py` 注册，新增端点自动生效。 |
