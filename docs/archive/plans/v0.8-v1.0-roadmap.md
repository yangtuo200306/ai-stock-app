# AI Stock App v0.8 - v1.0 后续版本路线计划

## 1. 文档定位

本文档用于记录 AI Stock App 在 v0.7 完成后的后续三个版本方向。

当前项目已完成：

- 账号系统；
- 自选股；
- AI 问股；
- 最小多轮追问；
- 股票简称识别；
- 技术指标增强；
- 问股历史详情；
- 记录系统；
- 未登录态引导；
- 新建会话；
- 继续追问；
- 自选摘要；
- 报告类型；
- API 错误结构初步统一。

后续重点不再继续堆叠大功能，而是逐步从“本地可用的学习项目”收口到“可以自己部署、自用测试的小产品”。

## 2. 后续总体路线

后续三个版本建议为：

```text
v0.8：页面体验与公共 UI 基础
v0.9：状态管理与稳定性
v1.0：部署准备与自用测试版
```

三个版本分别解决的问题：

```text
v0.8 解决“看起来和用起来不像产品”
v0.9 解决“数据状态不像正式 App”
v1.0 解决“只能本地跑，不能长期自用”
```

## 3. v0.8：页面体验与公共 UI 基础

### 3.1 版本目标

把 App 从“开发验证页面”升级为“用户能看懂、能顺着用”的产品界面。

v0.8 的重点不是单纯美化，而是：

```text
页面体验定型 + 公共 UI 基础 + 状态展示统一
```

### 3.2 主要方向

1. 抽取最小公共 UI 组件；
2. 抽取主题常量；
3. 抽取展示工具函数；
4. 优化四个核心 Tab；
5. 统一加载态、空状态、错误态和未登录态；
6. 优化 AI 长回答阅读体验。

### 3.3 建议抽取内容

建议新增轻量结构：

```text
mobile-app/src/components/
  AppButton.tsx
  AppCard.tsx
  StateView.tsx
  LoginRequiredView.tsx
  MessageBubble.tsx
  MetricRow.tsx

mobile-app/src/theme/
  colors.ts
  spacing.ts
  typography.ts

mobile-app/src/utils/
  recordDisplay.ts
  taskStatusDisplay.ts
  stockDisplay.ts
```

这些内容用于解决当前页面中重复的按钮、卡片、状态页、登录引导、消息气泡、指标行、颜色和展示文案问题。

### 3.4 页面优化顺序

建议按以下顺序推进：

```text
问股页 -> 自选页 -> 记录页 / 记录详情 -> 我的页 -> 全局收口
```

原因：

1. 问股页是产品核心入口，当前长回答和聊天体验压力最大；
2. 自选页连接“问 AI”和“生成报告”，是另一核心入口；
3. 记录页承接问股和报告，需要优化历史复看体验；
4. 我的页主要是账号、配置和应用信息，可以后置整理；
5. 最后统一检查视觉、状态和交互一致性。

### 3.5 v0.8 不做范围

v0.8 暂不做：

1. 大规模状态管理重构；
2. Redux / Zustand / React Query 等状态库引入；
3. 服务器部署；
4. 深色模式；
5. 股票搜索；
6. 历史搜索；
7. 分享功能；
8. 消息删除、编辑、重试；
9. 复杂 AI Agent 能力；
10. 大规模后端接口重构。

### 3.6 v0.8 完成效果

v0.8 完成后，用户应该能明显感受到：

1. 每个页面知道是干什么的；
2. 每个页面知道下一步该点哪里；
3. 空状态能引导用户开始操作；
4. 错误状态能提示用户如何处理；
5. AI 长回答更容易阅读；
6. 自选、问股、记录、我的四个 Tab 风格统一；
7. 代码中重复 UI 明显减少。

## 4. v0.9：状态管理与稳定性

### 4.1 版本目标

解决页面之间数据不同步、旧状态残留、刷新混乱、登录切换体验不稳定等问题。

v0.9 的重点是：

```text
业务状态统一 + 数据同步 + 登录状态稳定
```

### 4.2 当前状态管理问题

当前项目主要是：

```text
每个页面自己 useState
每个页面自己请求接口
每个页面自己处理 loading / error
```

可能带来的用户问题：

1. 问股成功后，记录页不一定马上更新；
2. 问股页加入自选后，自选页不一定马上同步；
3. 退出登录后，旧页面数据可能短暂残留；
4. 登录新账号后，可能短暂看到旧账号数据；
5. token 失效后，用户不知道需要重新登录；
6. loading / error 表现不一致；
7. 用户需要手动刷新较多；
8. 问股会话恢复不完整。

### 4.3 v0.9 建议方向

v0.9 可以优先考虑：

```text
contexts/
  WatchlistContext.tsx
  RecordsContext.tsx
```

或先抽页面级 hooks：

```text
hooks/
  useWatchlist.ts
  useRecords.ts
  useAskSession.ts
```

具体采用 Context 还是 hooks，可在 v0.9 计划阶段结合 v0.8 后的代码结构再决定。

### 4.4 v0.9 重点任务

1. 问股成功后刷新记录状态；
2. 加入自选后刷新自选状态；
3. 生成报告后刷新记录状态；
4. 退出登录时清空业务数据；
5. 切换账号时避免旧账号数据残留；
6. token 失效时清除 token 并提示重新登录；
7. API client 保留结构化错误信息；
8. 统一刷新和错误反馈体验；
9. 梳理问股会话恢复边界。

### 4.5 v0.9 不做范围

v0.9 暂不做：

1. 大规模 UI 改版；
2. 复杂 AI 能力扩展；
3. 正式服务器部署；
4. 数据库大迁移；
5. 股票搜索、分享、历史搜索等新功能。

### 4.6 v0.9 完成效果

v0.9 完成后，用户应该能感受到：

1. 操作后相关页面自动同步；
2. 退出登录后不再看到旧数据；
3. 登录切换更稳定；
4. token 失效时知道需要重新登录；
5. 页面刷新和错误反馈更像正式 App。

## 5. v1.0：部署准备与自用测试版

### 5.1 版本目标

让项目可以部署到服务器，并支持用户自己长期使用测试。

v1.0 的重点不是继续增加功能，而是：

```text
能稳定运行
能配置
能排错
能备份
能从公网访问
```

### 5.2 后端部署准备

v1.0 需要梳理 FastAPI 后端部署方式，包括：

1. 服务启动方式；
2. 服务器端口；
3. CORS 配置；
4. 环境变量；
5. AI API Key 管理；
6. SQLite 数据库路径；
7. 日志查看；
8. 健康检查；
9. 后台运行和重启策略。

### 5.3 数据库策略

当前项目使用 SQLite。

自用测试阶段可以继续使用 SQLite，但需要明确：

1. SQLite 文件放在哪里；
2. 如何备份；
3. 服务器重启后数据是否保留；
4. 是否需要定期复制备份；
5. 后续是否需要迁移到 PostgreSQL。

v1.0 阶段暂不强制迁移数据库。

### 5.4 前端访问方式

v1.0 可以优先考虑 Web 自用部署：

```text
Expo Web 构建 -> 部署为网页 -> 手机浏览器访问
```

同时保留移动端后端地址配置能力，方便继续用 Expo 或移动端调试。

### 5.5 基础安全要求

自用测试阶段至少需要注意：

1. API Key 不写死到代码中；
2. API Key 不提交到 Git；
3. 服务器使用环境变量；
4. 密码继续使用哈希保存；
5. token 不打印到日志；
6. CORS 不随意开放到危险范围；
7. 部署前检查敏感文件。

### 5.6 部署文档

到 v1.0 阶段，可以在用户确认后生成部署文档，内容包括：

1. 服务器准备；
2. 后端启动；
3. 前端构建；
4. 环境变量配置；
5. 数据库备份；
6. 常见问题排查；
7. 部署验收流程。

### 5.7 v1.0 完成效果

v1.0 完成后，项目应该达到：

1. 后端可在服务器稳定运行；
2. 前端可通过浏览器或移动端访问；
3. 登录、自选、问股、记录核心链路可用；
4. 数据不会因重启轻易丢失；
5. 出错时有基本日志可查；
6. 用户可以自己长期使用测试。

## 6. 暂不纳入后续三个版本的大功能

以下功能建议 v1.0 之后再考虑：

1. 股票搜索；
2. 分享分析结果；
3. AI 回答复制；
4. 历史记录搜索；
5. 自选排序；
6. 手动刷新行情；
7. 深色模式；
8. 消息删除、编辑、重试；
9. 推送提醒；
10. 持仓管理；
11. 交易记录；
12. 回测；
13. 多市场支持；
14. 复杂 Agent 技能系统；
15. 专业投研工作台。

## 7. 总结

后续路线应从功能扩展转向产品收口：

```text
v0.8：先让用户看得懂、愿意用
v0.9：再让数据状态稳定、同步自然
v1.0：最后让项目能部署、自用测试
```

核心原则：

1. 不再盲目堆功能；
2. 每个版本只解决一类主要问题；
3. 保持学习节奏；
4. 优先完成可运行、可验证、可维护的小产品。
