# v0.6 AI 助手能力增强计划

## 1. 版本目标

把当前“单次问股接口展示”升级为“最小 AI 股票助手”。

最终 v0.6 完成后应支持：

1. 股票简称识别；
2. 最小多轮追问；
3. 技术指标增强；
4. AI 回答结构边界明确；
5. 问股历史详情增强。

## 2. 版本拆分

### 2.1 v0.6 阶段 1：AI 助手底座

目标：先让问股具备“会话”和“历史消息”。

核心结果：

```text
第一次问股 -> 创建会话 -> 保存用户问题和助手回答
继续追问 -> 复用 session_id -> 保存新一轮问答
记录详情 -> 能看到完整会话
```

### 2.2 v0.6 阶段 2：AI 助手增强

目标：让助手更自然、更有分析材料。

核心结果：

```text
支持常用股票简称
技术指标增加 RSI 和成交量变化
AI 回答使用最近几轮上下文
问股页展示更接近聊天
```

## 3. 阶段 1：AI 助手底座详细计划

### 3.1 数据库调整

涉及文件：

- `backend/app/database.py`

新增两张表。

#### 3.1.1 ask_sessions 表

用途：保存一次问股会话。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PRIMARY KEY AUTOINCREMENT | 内部主键 |
| session_id | TEXT UNIQUE NOT NULL | 前后端使用的会话 ID |
| user_id | TEXT NOT NULL | 当前用户 |
| stock_code | TEXT NOT NULL | 会话绑定股票代码 |
| stock_name | TEXT NOT NULL | 股票名称 |
| title | TEXT NOT NULL | 会话标题 |
| summary | TEXT NOT NULL | 会话摘要 |
| last_question | TEXT | 最近一次用户问题 |
| last_answer | TEXT | 最近一次助手回答 |
| answer_type | TEXT | 最近一次回答来源，ai / rule |
| ai_status | TEXT | 最近一次模型状态，ok / fallback |
| model | TEXT | 模型名 |
| created_at | TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP | 更新时间 |

边界：

- 一个会话只绑定一只股票；
- v0.6 不做一个会话里切换多只股票；
- `session_id` 使用 uuid 字符串；
- `user_id + session_id` 查询时必须一起校验，防止跨用户访问。

#### 3.1.2 ask_messages 表

用途：保存会话中的多轮消息。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PRIMARY KEY AUTOINCREMENT | 内部主键 |
| session_id | TEXT NOT NULL | 所属会话 |
| user_id | TEXT NOT NULL | 当前用户 |
| role | TEXT NOT NULL | user / assistant |
| content | TEXT NOT NULL | 消息内容 |
| answer_type | TEXT | assistant 消息使用 |
| ai_status | TEXT | assistant 消息使用 |
| model | TEXT | assistant 消息使用 |
| created_at | TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP | 创建时间 |

边界：

- 用户消息和助手消息分两行保存；
- `role=user` 时 `answer_type`、`ai_status`、`model` 可以为空；
- `role=assistant` 时保存回答来源和 AI 状态；
- v0.6 不做消息删除；
- v0.6 不做分页，单会话消息数量先按学习项目接受。

### 3.2 后端问股接口调整

涉及文件：

- `backend/app/api/ask.py`
- `backend/app/services/llm_client.py`

#### 3.2.1 AskCreate 调整

当前字段：

- `question`
- `stock_code`

计划增加：

| 字段 | 类型 | 说明 |
|---|---|---|
| session_id | string / null | 继续追问时传入 |
| question | string / null | 用户问题 |
| stock_code | string / null | 直接按代码问股 |

规则：

1. 如果没有 `session_id`：
   - 按当前逻辑从问题或 `stock_code` 中识别股票；
   - 创建新会话。
2. 如果有 `session_id`：
   - 后端根据 `session_id + user_id` 找到会话；
   - 使用会话中的 `stock_code`；
   - 本轮问题作为追问；
   - 不允许追问中切换到另一只股票。
3. 如果 `session_id` 不存在或不属于当前用户：
   - 返回“会话不存在”。

#### 3.2.2 AskResponse 调整

当前响应仍保留原字段，避免前端大面积重构。

计划新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| session_id | string | 当前会话 ID |
| message_id | number / null | 本轮 assistant 消息 ID |
| is_new_session | boolean | 是否新建会话 |
| conversation_title | string | 会话标题 |

边界说明：

- `answer_type`：表示当前展示回答来源，取值 `ai` / `rule`。
- `ai_status`：表示本轮大模型调用状态，取值 `ok` / `fallback`。
- `model`：表示本轮尝试使用的模型名。
- v0.6 暂不把 `stock`、`market`、`technicals`、`ai` 全部分组重构，避免范围过大。
- 分组式响应结构留到 v0.7 功能定型或 v0.8 UI 优化前统一处理。

#### 3.2.3 新增会话写入逻辑

在 `backend/app/api/ask.py` 中增加内部函数职责：

1. 创建 ask session；
2. 查询 ask session；
3. 写入 user message；
4. 写入 assistant message；
5. 更新 ask session 的最近问题、最近回答、摘要、更新时间；
6. 写入或更新 records 中的问股记录。

命名建议：

- `_create_ask_session`
- `_get_ask_session`
- `_write_ask_message`
- `_update_ask_session`
- `_write_or_update_ask_record`

注意：

- 这些函数只在 `backend/app/api/ask.py` 内部使用；
- 暂不抽新 service，避免过度设计；
- 如果函数明显过长，后续 v0.7 再考虑拆分。

### 3.3 records 与问股会话关系调整

涉及文件：

- `backend/app/database.py`
- `backend/app/api/ask.py`
- `backend/app/api/records.py`

#### 3.3.1 records 表扩展

计划给 `records` 增加：

| 字段 | 类型 | 说明 |
|---|---|---|
| session_id | TEXT | 问股会话 ID |

规则：

- 问股记录 `record_type=ask` 时保存 `session_id`；
- 自选分析记录 `record_type=analysis` 时 `session_id` 为空；
- 使用 `ensure_column` 做兼容迁移。

#### 3.3.2 问股记录写入策略

当前每次问股都会新建一条记录。

v0.6 阶段 1 调整为：

1. 新建会话时：
   - 新增一条 `records`；
   - `record_type=ask`；
   - `session_id` 指向 ask session；
   - `summary` 使用首次回答摘要。
2. 继续追问时：
   - 不新增 records；
   - 更新同一个 `session_id` 对应的 records；
   - `summary` 更新为最近回答摘要；
   - `question` 更新为最近问题；
   - `answer` 更新为最近回答；
   - `metadata_json` 更新最新行情和 AI 状态。

原因：

- 记录列表展示“会话”更合理；
- 否则每次追问都会刷出多条记录，历史页会很乱。

### 3.4 records 接口增强

涉及文件：

- `backend/app/api/records.py`

#### 3.4.1 列表接口

`GET /api/records` 需要返回 `session_id`。

前端记录列表不需要展示所有消息，只展示会话摘要。

#### 3.4.2 详情接口

`GET /api/records/{record_id}` 增强规则：

1. 如果记录没有 `session_id`：
   - 按当前逻辑返回单条详情。
2. 如果记录有 `session_id`：
   - 额外查询 `ask_messages`；
   - 返回 `messages` 数组。

`messages` 字段结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | number | 消息 ID |
| role | string | user / assistant |
| content | string | 内容 |
| answer_type | string / null | assistant 消息来源 |
| ai_status | string / null | AI 状态 |
| model | string / null | 模型 |
| created_at | string | 时间 |

边界：

- 只返回当前用户自己的消息；
- 查询时必须带 `user_id`；
- 不做跨用户会话读取。

### 3.5 AI 上下文增强

涉及文件：

- `backend/app/services/llm_client.py`
- `backend/app/api/ask.py`

#### 3.5.1 ask_llm 入参调整

计划让 `ask_llm` 支持可选历史上下文。

新增参数概念：

| 参数 | 说明 |
|---|---|
| conversation_messages | 最近几轮问答 |

规则：

- 只传最近 4 条消息，避免 prompt 过长；
- 包含 user 和 assistant；
- 不做长期记忆；
- 不做上下文压缩。

#### 3.5.2 prompt 边界

Prompt 需要明确：

1. 当前股票固定为本会话股票；
2. 只能基于当前行情、技术指标和最近上下文回答；
3. 如果用户问到非当前股票，提示“当前会话围绕某股票，如需分析其他股票请新开问题”；
4. 不给确定性买卖指令；
5. 必须保留免责声明。

## 4. 阶段 2：AI 助手增强详细计划

### 4.1 股票简称识别

涉及文件：

- 新增 `backend/app/services/stock_resolver.py`
- 修改 `backend/app/api/ask.py`

#### 4.1.1 新增 stock_resolver 服务

职责：

1. 从文本中优先识别 6 位股票代码；
2. 如果没有代码，再识别常用股票名称；
3. 返回标准股票代码；
4. 如果无法识别，返回空；
5. 如果未来出现歧义，返回提示。

#### 4.1.2 内置最小股票映射

v0.6 先支持少量常用 A 股：

| 名称 | 代码 |
|---|---|
| 平安银行 | 000001 |
| 万科A | 000002 |
| 贵州茅台 | 600519 |
| 招商银行 | 600036 |
| 宁德时代 | 300750 |
| 比亚迪 | 002594 |
| 五粮液 | 000858 |
| 中国平安 | 601318 |
| 工商银行 | 601398 |
| 东方财富 | 300059 |

边界：

- 不做全市场搜索；
- 不做拼音；
- 不做模糊匹配；
- 不做港股、美股；
- 不做歧义复杂处理。

#### 4.1.3 问股接口使用 resolver

`backend/app/api/ask.py` 中原 `_extract_stock_code` 替换为“解析股票输入”的内部调用。

识别顺序：

```text
明确 stock_code 参数
-> question 中 6 位代码
-> question 中内置股票名称
-> 失败提示用户输入 6 位 A 股代码或已支持的股票名称
```

### 4.2 技术指标增强

涉及文件：

- `backend/app/services/technical_indicators.py`
- `backend/app/services/report_builder.py`
- `backend/app/services/llm_client.py`
- `mobile-app/src/types/index.ts`
- `mobile-app/src/screens/AskScreen.tsx`
- `mobile-app/src/screens/ReportDetailScreen.tsx`

#### 4.2.1 新增 RSI

在 `backend/app/services/technical_indicators.py` 增加 RSI 计算。

规划：

| 字段 | 说明 |
|---|---|
| rsi6 | 6 日 RSI |
| rsi12 | 12 日 RSI |

边界：

- 数据不足时返回 `0.0`；
- RSI 结果保留两位小数；
- 暂不实现复杂多周期解释。

#### 4.2.2 新增成交量变化

基于 `StockDailyPrice.volume` 计算：

| 字段 | 说明 |
|---|---|
| volume_ratio | 最近一日成交量 / 最近 5 日平均成交量 |
| volume_signal | 放量 / 缩量 / 平稳 |

规则：

- `volume_ratio >= 1.5`：放量；
- `volume_ratio <= 0.7`：缩量；
- 其他：平稳；
- 数据不足或平均成交量为 0 时返回默认值。

#### 4.2.3 报告构建增强

`backend/app/services/report_builder.py` 需要：

1. 把 RSI 写入 `indicators`；
2. 把成交量变化写入 `indicators`；
3. `score_reasons` 增加 RSI / 成交量相关原因；
4. `risks` 增加 RSI 过热或成交量异常提示；
5. 评分逻辑可以轻微使用 RSI 和成交量，但不做复杂量化策略。

建议评分边界：

- RSI 偏高时轻微扣分；
- RSI 偏低时不直接加大分，只提示风险；
- 放量上涨可轻微加分；
- 放量下跌或大跌可增加风险提示；
- 不输出“买入/卖出”确定性建议。

### 4.3 AI Prompt 增强

涉及文件：

- `backend/app/services/llm_client.py`

Prompt 增加内容：

1. RSI 指标；
2. 成交量变化；
3. 最近几轮对话；
4. 当前会话股票边界；
5. 回答追问时先回应用户当前问题，再补充指标解释。

### 4.4 前端问股页增强

涉及文件：

- `mobile-app/src/screens/AskScreen.tsx`
- `mobile-app/src/types/index.ts`

#### 4.4.1 状态结构调整

当前 `AskScreen.tsx` 只有单个 `result`。

计划调整为：

| 状态 | 说明 |
|---|---|
| sessionId | 当前会话 ID |
| messages | 当前页面消息列表 |
| latestResult | 最近一次问股结果 |
| question | 输入框内容 |
| loading | 提交状态 |
| error | 错误信息 |

#### 4.4.2 展示方式

v0.6 不做大 UI 美化，但做基础聊天形态：

1. 用户问题显示为一条消息；
2. AI 回答显示为一条消息；
3. 顶部或回答下方显示当前股票、价格、涨跌幅、评分；
4. 技术指标区域展示：
   - MA5 / MA10 / MA20；
   - RSI6 / RSI12；
   - 成交量信号；
5. 保留“加入自选”按钮；
6. 用户继续输入问题时，自动带上当前 `session_id`。

边界：

- 不做流式输出；
- 不做消息删除；
- 不做复制；
- 不做重新生成；
- 不做复杂样式重构。

### 4.5 记录详情页增强

涉及文件：

- `mobile-app/src/screens/RecordDetailScreen.tsx`

#### 4.5.1 类型增强

`RecordDetail` 增加：

| 字段 | 类型 |
|---|---|
| session_id | string / null |
| messages | AskMessage[] / optional |

新增前端类型：

| 类型 | 字段 |
|---|---|
| AskMessage | id、role、content、answer_type、ai_status、model、created_at |

#### 4.5.2 页面展示规则

如果 `messages` 存在且不为空：

1. 展示完整问答消息；
2. user 消息标注“我”；
3. assistant 消息标注“AI 回答”或“规则回退回答”；
4. 保留当前股票摘要、价格、涨跌幅、评分、趋势、建议；
5. 保留创建时间。

如果没有 `messages`：

- 继续使用当前单条问股详情展示逻辑，兼容旧数据。

## 5. 不纳入 v0.6 的内容

以下内容不做：

1. 全市场股票搜索；
2. 股票简称模糊匹配；
3. 港股、美股；
4. MACD 完整实现；
5. 复杂 Agent 技能系统；
6. 工具调用型 AI；
7. 流式输出；
8. 长上下文压缩；
9. 消息删除、编辑、重试；
10. API 错误结构统一；
11. 大规模 UI/UX 美化；
12. 用户旧数据迁移；
13. token 过期和刷新；
14. 记录模块完整重构。

## 6. 验收标准

### 6.1 阶段 1 验收

1. 登录用户可以输入 `000001 怎么看？`。
2. 后端返回 `session_id`。
3. 前端继续追问时携带同一个 `session_id`。
4. 后端保存同一个会话下的多轮 user / assistant 消息。
5. 记录列表只出现一条对应问股会话记录。
6. 记录详情能展示完整多轮问答。
7. 用户 A 不能访问用户 B 的会话和消息。
8. AI 失败时仍使用规则回退，并保存 `answer_type=rule`、`ai_status=fallback`。

### 6.2 阶段 2 验收

1. 输入“平安银行怎么看？”能识别为 `000001`。
2. 输入“贵州茅台怎么看？”能识别为 `600519`。
3. 输入未知股票名称时提示输入 6 位 A 股代码或已支持名称。
4. 问股结果包含 RSI。
5. 问股结果包含成交量变化。
6. AI 回答能参考最近几轮上下文。
7. 问股页能展示多轮消息。
8. 记录详情能展示多轮历史。
9. 原来的 6 位代码问股仍可用。
10. 自选分析、报告详情、记录列表不被破坏。

## 7. 学习重点

v0.6 学习重点：

1. 为什么聊天系统需要 session 和 message 两层结构；
2. 为什么记录列表更适合展示“会话摘要”，不是每条消息；
3. 前端如何从单结果页面演进成简单聊天页面；
4. AI 上下文为什么不能无限传；
5. 股票简称识别为什么先做小词典，而不是全市场搜索；
6. RSI 和成交量变化的基本含义；
7. API 响应结构为什么要逐步建立边界；
8. 用户隔离为什么要贯穿会话、消息和记录查询。
