# AI Stock App v0.9 版本计划：状态管理与稳定性最小版

## 1. 版本定位

v0.9 定位为：

```text
状态管理与稳定性最小版
```

本版本不追求完整缓存系统，也不引入 Zustand / React Query，而是先解决当前移动端最影响体验的状态问题：

```text
登录状态稳定 + token 失效处理 + 页面间刷新同步 + 任务状态刷新
```

v0.9 是后续引入 Zustand、React Query 或更完整缓存系统前的地基版本。

## 2. 版本目标

v0.9 需要完成以下目标：

1. 登录、注册、退出登录后，业务页面状态稳定刷新。
2. 退出登录后不再残留旧账号的自选、记录、问股结果和任务状态。
3. 切换账号后不短暂显示上一个账号的数据。
4. token 运行中失效时，能统一清除本地 token，并提示用户重新登录。
5. 问股成功后，记录页能刷新。
6. 问股页加入自选后，自选页能刷新。
7. 自选页添加、删除、创建分析任务后，相关页面能同步刷新。
8. 任务状态页支持轻量轮询，减少用户手动刷新。
9. 任务完成后，自选摘要和记录列表能刷新。
10. 减少导航类型中的 `any`。
11. 保持 v0.8 UI 体验不回退。
12. 为后续 Zustand / 缓存层引入留下清晰状态边界。

## 3. 本版本不做范围

v0.9 明确不做：

1. 不引入 Zustand。
2. 不引入 React Query。
3. 不做完整请求缓存。
4. 不做离线缓存。
5. 不做 WebSocket。
6. 不做 SSE。
7. 不做后台任务队列重构。
8. 不大改后端接口。
9. 不大改数据库结构。
10. 不做 UI 大改版。
11. 不新增股票搜索、历史搜索、分享、AI 回答复制等功能。
12. 不 push，v0.9 完成并确认后再决定是否和 v0.8 一起 push。

## 4. 分阶段实施方案

v0.9 在一个版本内分 4 个阶段执行。

## 5. 阶段 1：状态基础层

### 5.1 阶段目标

先建立统一状态地基：

```text
结构化 API 错误 + 鉴权状态清理 + 数据刷新信号
```

本阶段不优先改复杂页面逻辑，先保证基础能力稳定。

### 5.2 涉及文件

现有文件：

- `mobile-app/src/api/client.ts`
- `mobile-app/src/contexts/AuthContext.tsx`
- `mobile-app/src/navigation/AppNavigator.tsx`

新增文件：

- `mobile-app/src/contexts/DataRefreshContext.tsx`
- `mobile-app/src/hooks/useApiErrorHandler.ts`

### 5.3 具体计划

#### 5.3.1 结构化 API 错误

修改 `mobile-app/src/api/client.ts`。

新增统一 API 错误结构，至少包含：

1. `message`
2. `status`
3. `errorCode`
4. `raw`

统一解析当前后端错误格式：

1. `detail.message`
2. `detail.error_code`
3. `detail` 字符串
4. `message`
5. 非 JSON 响应兜底为普通 HTTP 错误提示

`apiGet`、`apiPost`、`apiDelete` 必须共用同一套错误解析逻辑。

#### 5.3.2 增强 AuthContext

修改 `mobile-app/src/contexts/AuthContext.tsx`。

新增或整理以下能力：

1. 内部统一清理鉴权状态。
2. token 恢复失败时清空本地 token 和内存用户状态。
3. 暴露 token 运行中失效时可调用的清理方法。
4. 登录成功后设置 token、userId、username。
5. 注册成功后设置 token、userId、username。
6. 退出登录时，即使后端 logout 请求失败，也清除本地 token 和内存用户状态。
7. `isLoggedIn` 继续基于 token，但 token 清除后必须稳定切换到未登录态。

#### 5.3.3 新增 DataRefreshContext

新增 `mobile-app/src/contexts/DataRefreshContext.tsx`。

职责：

1. 不保存真实业务数据。
2. 只保存刷新信号版本号。
3. 提供以下版本号：
   - `watchlistVersion`
   - `recordsVersion`
   - `tasksVersion`
4. 提供以下通知函数：
   - `notifyWatchlistChanged`
   - `notifyRecordsChanged`
   - `notifyTasksChanged`
   - `notifyAllDataChanged`

设计边界：

```text
DataRefreshContext 只负责“通知该刷新”
不负责“缓存数据”
不负责“请求接口”
不负责“业务判断”
```

#### 5.3.4 挂载 DataRefreshProvider

修改 `mobile-app/src/navigation/AppNavigator.tsx`。

目标：

1. 保留 `AuthProvider`。
2. 新增 `DataRefreshProvider`。
3. 保持当前底部 Tab：
   - 自选
   - 问股
   - 记录
   - 我的
4. 不改导航层级和页面结构。

#### 5.3.5 新增 useApiErrorHandler

新增 `mobile-app/src/hooks/useApiErrorHandler.ts`。

职责：

1. 接收页面 catch 到的 `unknown` 错误。
2. 判断是否为：
   - HTTP 401
   - `INVALID_TOKEN`
   - `INVALID_TOKEN_FORMAT`
3. 如果是 token 失效：
   - 调用 AuthContext 的鉴权清理方法；
   - 调用 `notifyAllDataChanged`；
   - 返回统一提示：“登录状态已失效，请重新登录”。
4. 如果不是 token 问题：
   - 返回结构化错误中的 message；
   - 或返回页面传入的默认错误文案。

## 6. 阶段 2：核心页面同步

### 6.1 阶段目标

解决当前最明显的页面间不同步：

1. 问股成功后记录页刷新。
2. 问股加入自选后自选页刷新。
3. 自选添加 / 删除后自选页刷新。
4. 登录切换后清空旧业务状态。

### 6.2 涉及文件

- `mobile-app/src/screens/AskScreen.tsx`
- `mobile-app/src/screens/WatchlistScreen.tsx`
- `mobile-app/src/screens/RecordListScreen.tsx`

- `mobile-app/src/screens/LoginScreen.tsx`

### 6.3 具体计划

#### 6.3.1 问股页同步

修改 `mobile-app/src/screens/AskScreen.tsx`。

调整内容：

1. 接入 `DataRefreshContext`。
2. 接入 `useApiErrorHandler`。
3. 问股成功后调用 `notifyRecordsChanged`。
4. 问股成功后，如会影响自选摘要，则调用 `notifyWatchlistChanged`。
5. 加入自选成功后调用 `notifyWatchlistChanged`。
6. 登录状态变为未登录时，清空：
   - `sessionId`
   - `messages`
   - `latestResult`
   - `question`
   - `error`
7. 保持当前 UI、MessageBubble、MetricRow、AppButton 不变。
8. 不新增消息删除、复制、重试、流式输出。

#### 6.3.2 自选页同步

修改 `mobile-app/src/screens/WatchlistScreen.tsx`。

调整内容：

1. 接入 `DataRefreshContext`。
2. 接入 `useApiErrorHandler`。
3. `watchlistVersion` 变化时重新加载自选。
4. `tasksVersion` 变化时重新加载任务状态。
5. 添加自选成功后调用 `notifyWatchlistChanged`。
6. 删除自选成功后调用：
   - `notifyWatchlistChanged`
   - `notifyTasksChanged`
7. 创建分析任务成功后调用 `notifyTasksChanged`。
8. 登录状态变为未登录时，清空：
   - `stocks`
   - `taskStatuses`
   - `loadError`
   - `newCode`
   - `newName`
9. 保留 `readTaskIds()` 的 JSON 解析保护。
10. 不改自选页 UI 结构。

#### 6.3.3 记录列表同步

修改 `mobile-app/src/screens/RecordListScreen.tsx`。

调整内容：

1. 接入 `DataRefreshContext`。
2. 接入 `useApiErrorHandler`。
3. `recordsVersion` 变化时重新加载记录。
4. 登录状态变为未登录时，清空：
   - `items`
   - `error`
   - 合理处理 `loading`
5. 保持 StateView、AppCard、recordDisplay、stockDisplay 使用方式不变。

#### 6.3.4 我的页退出登录联动

修改 `mobile-app/src/screens/MineScreen.tsx`。

调整内容：

1. 新增页面内 `handleLogout`。
2. 退出登录按钮调用 `handleLogout`，不再裸调用 `logout`。
3. `handleLogout` 调用 AuthContext 的 `logout`。
4. 退出完成后触发 `notifyAllDataChanged`，或由 AuthContext 统一触发。
5. 后端地址配置、测试连接逻辑不变。

#### 6.3.5 登录页登录成功联动

修改 `mobile-app/src/screens/LoginScreen.tsx`。

调整内容：

1. 登录成功后调用 `notifyAllDataChanged`。
2. 注册成功后调用 `notifyAllDataChanged`。
3. 保持登录 / 注册切换逻辑不变。
4. 保持现有输入校验不变。
5. 登录成功后仍返回上一页。

## 7. 阶段 3：任务状态稳定性

### 7.1 阶段目标

把任务状态从“只靠手动刷新”提升为：

```text
手动刷新 + 轻量轮询 + 完成后联动刷新
```

### 7.2 涉及文件

- `mobile-app/src/screens/TaskStatusScreen.tsx`
- `mobile-app/src/screens/WatchlistScreen.tsx`

### 7.3 具体计划

#### 7.3.1 任务状态页轻量轮询

修改 `mobile-app/src/screens/TaskStatusScreen.tsx`。

调整内容：

1. 接入 `DataRefreshContext`。
2. 接入 `useApiErrorHandler`。
3. 保留“刷新状态”按钮。
4. 增加轻量轮询。
5. 仅当满足以下条件时轮询：
   - 已登录；
   - 鉴权恢复完成；
   - 当前任务状态是 `pending` 或 `running`。
6. 当任务状态为以下状态时停止轮询：
   - `completed`
   - `failed`
7. 每次任务状态获取成功后调用 `notifyTasksChanged`。
8. 任务变为 `completed` 且存在 `report_id` 时调用：
   - `notifyRecordsChanged`
   - `notifyWatchlistChanged`
   - `notifyTasksChanged`
9. 登录状态变为未登录时清空：
   - `task`
   - `error`
10. 不引入 SSE，不引入 WebSocket，不改后端任务模型。

#### 7.3.2 自选页任务状态响应

继续调整 `mobile-app/src/screens/WatchlistScreen.tsx`。

调整内容：

1. 自选页响应 `tasksVersion`。
2. `tasksVersion` 变化时重新读取本地 taskId 并请求服务端任务状态。
3. 本地 task 状态仍以 AsyncStorage 的 `stockTaskIds` 为索引。
4. 不把 task 状态做成完整缓存。
5. 不改后端任务接口。

## 9. 后端处理策略

v0.9 默认不改后端。

当前后端已具备本版本需要的基础：

1. 受保护接口通过 token 鉴权。
2. token 无效返回 401。
3. 401 detail 中已有 `error_code` 和 `message`。
4. 任务查询按 `task_id + user_id` 查询。
5. 分析任务当前同步完成或失败。

除非执行阶段发现前端无法可靠识别 token 失效，否则不改后端。

## 11. 风险与控制

### 11.1 风险：刷新信号重复触发

控制方式：

1. v0.9 只使用版本号加一。
2. 不在 DataRefreshContext 中直接请求接口。
3. 页面自行判断是否已登录后再请求。

### 11.2 风险：token 失效处理造成重复弹错

控制方式：

1. `useApiErrorHandler` 返回统一文案。
2. 页面只负责展示当前页面已有错误区域或 Alert。
3. 不在 API client 里直接弹窗。

### 11.3 风险：任务轮询不停止

控制方式：

1. 只在 `pending/running` 时轮询。
2. `completed/failed` 后停止。
3. 页面卸载时清理 interval。

### 11.4 风险：为了减少 any 扩大导航重构

控制方式：

1. 只做低风险类型收口。
2. 不重构导航结构。
3. 高成本类型问题记录到后续，不强行处理。

### 11.5 风险：状态同步做成伪缓存

控制方式：

1. v0.9 不缓存业务数据。
2. 页面仍从后端重新加载真实数据。
3. DataRefreshContext 只做刷新通知。
