# v0.4 用户边界 + 问股闭环详细计划

## 1. 版本目标

v0.4 的目标是让 AI Stock App 从“能问股、能看报告”升级为“用户行为可归属、问股和自选形成闭环”。

本版本重点不是 UI 精修，也不做真正注册登录，而是先搭好三个基础边界：

1. 所有核心数据预留默认 `user_id`。
2. 新增最小记录体系，承接问股、自选分析和报告。
3. 打通问股、自选、记录三个模块之间的基础联动。

## 2. 本版本范围

### 2.1 做

1. 后端预留默认 `user_id`。
2. `stocks` 自选表增加 `user_id`。
3. `reports` 报告表增加 `user_id`。
4. `analysis_tasks` 分析任务表增加 `user_id`。
5. 新增 `records` 记录表。
6. 新增记录接口：
   - `GET /api/records`
   - `GET /api/records/{record_id}`
7. 问股成功后写入记录。
8. 自选分析成功生成报告后写入记录。
9. 底部 Tab 从“报告”调整为“记录”。
10. 原历史报告页改造成记录列表页。
11. 问股结果支持加入自选。
12. 自选页支持快速问 AI。
13. 隐藏或改造 `llm_error`，不再直接作为前端依赖字段。
14. 问股页做最小滚动修复。

### 2.2 不做

1. 不做真正注册、登录、退出登录。
2. 不做 token。
3. 不做多用户切换。
4. 不做多轮追问。
5. 不做股票简称识别。
6. 不做 UI/UX 大优化。
7. 不做复杂记录搜索、筛选、分页。
8. 不做旧数据复杂迁移工具。
9. 不删除原 `reports` 表，报告仍作为正式分析详情保存。

## 3. 默认用户边界设计

### 3.1 默认用户常量

在后端新增统一默认用户常量。

涉及文件：

- `backend/app/database.py`

计划新增：

```text
DEFAULT_USER_ID = "default_user"
```

使用原则：

1. v0.4 所有新增写入数据默认使用 `DEFAULT_USER_ID`。
2. 暂不从请求头、参数或登录态读取用户。
3. v0.5 做真实用户系统时，再把默认用户替换为登录用户。

### 3.2 需要绑定 user_id 的表

当前已有表：

- `stocks`
- `analysis_tasks`
- `reports`

v0.4 计划补充：

| 表 | 新增字段 | 默认值 |
|---|---|---|
| `stocks` | `user_id TEXT NOT NULL DEFAULT 'default_user'` | `default_user` |
| `analysis_tasks` | `user_id TEXT NOT NULL DEFAULT 'default_user'` | `default_user` |
| `reports` | `user_id TEXT NOT NULL DEFAULT 'default_user'` | `default_user` |

### 3.3 查询隔离规则

v0.4 所有查询先按默认用户过滤：

1. `GET /api/stocks` 只查 `user_id = DEFAULT_USER_ID`。
2. `POST /api/stocks` 写入 `user_id = DEFAULT_USER_ID`。
3. `DELETE /api/stocks/{code}` 只删除默认用户自己的自选。
4. `POST /api/analysis` 创建任务、报告、记录时写入默认用户。
5. `GET /api/analysis/{task_id}` 只查默认用户任务。
6. `GET /api/reports` 只查默认用户报告。
7. `GET /api/reports/{report_id}` 只查默认用户报告。
8. `GET /api/records` 只查默认用户记录。
9. `GET /api/records/{record_id}` 只查默认用户记录。

## 4. 记录体系最小版设计

### 4.1 新增 records 表

涉及文件：

- `backend/app/database.py`

新增表：

```text
records
```

字段设计：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | 记录 ID |
| `user_id` | TEXT NOT NULL | 用户 ID，v0.4 固定 default_user |
| `record_type` | TEXT NOT NULL | 记录类型 |
| `stock_code` | TEXT NOT NULL | 股票代码 |
| `stock_name` | TEXT NOT NULL | 股票名称 |
| `title` | TEXT NOT NULL | 记录标题 |
| `summary` | TEXT NOT NULL | 列表摘要 |
| `question` | TEXT | 问股问题，可为空 |
| `answer` | TEXT | 问股回答，可为空 |
| `answer_type` | TEXT | ai / rule，可为空 |
| `report_id` | INTEGER | 关联报告 ID，可为空 |
| `metadata_json` | TEXT NOT NULL | 附加信息 JSON |
| `created_at` | TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP | 创建时间 |

### 4.2 record_type 取值

v0.4 只使用两个类型：

| 类型 | 来源 | 说明 |
|---|---|---|
| `ask` | 问股成功 | 保存问题、回答、行情摘要 |
| `analysis` | 自选页点分析成功 | 保存分析任务生成的报告入口 |

保留但不实现：

- `report`
- `chat`

### 4.3 metadata_json 最小内容

问股记录：

```text
{
  "price": number,
  "change_pct": number,
  "score": number,
  "action": string,
  "trend": string,
  "model": string | null
}
```

分析记录：

```text
{
  "price": number,
  "change_pct": number | null,
  "score": number,
  "action": string,
  "trend": string,
  "task_id": string
}
```

## 5. 后端接口计划

### 5.1 新增 records API 文件

新增文件：

- `backend/app/api/records.py`

新增接口：

#### GET /api/records

返回记录列表。

返回结构：

```text
{
  "items": [
    {
      "id": number,
      "record_type": string,
      "stock_code": string,
      "stock_name": string,
      "title": string,
      "summary": string,
      "question": string | null,
      "answer_type": string | null,
      "report_id": number | null,
      "metadata": object,
      "created_at": string
    }
  ]
}
```

排序：

```text
ORDER BY id DESC
```

#### GET /api/records/{record_id}

返回单条记录详情。

返回结构：

```text
{
  "id": number,
  "record_type": string,
  "stock_code": string,
  "stock_name": string,
  "title": string,
  "summary": string,
  "question": string | null,
  "answer": string | null,
  "answer_type": string | null,
  "report_id": number | null,
  "metadata": object,
  "created_at": string
}
```

找不到时：

```text
{
  "message": "record not found",
  "record_id": number
}
```

### 5.3 stocks API 改造

涉及文件：

- `backend/app/api/stocks.py`

计划：

1. 导入 `DEFAULT_USER_ID`。
2. `GET /api/stocks` 增加默认用户过滤。
3. `POST /api/stocks` 插入 `user_id`。
4. 自选唯一性从全局 `code UNIQUE` 逐步调整为用户边界：
   - v0.4 最小方案：保留旧唯一约束不强改索引，只补 `user_id` 字段和查询条件。

### 5.4 ask API 改造

涉及文件：

- `backend/app/api/ask.py`

计划：

1. 导入 `json`。
2. 导入 `get_connection`。
3. 导入 `DEFAULT_USER_ID`。
4. `AskResponse` 中移除或隐藏 `llm_error` 对外返回。
5. 内部仍允许捕获 `LlmError`，但不把原始错误返回给前端。
6. 可新增返回字段：
   - `ai_status`
7. `ai_status` 取值：
   - `ok`
   - `fallback`
8. 当大模型成功：
   - `answer_type = "ai"`
   - `ai_status = "ok"`
9. 当大模型失败：
   - `answer_type = "rule"`
   - `ai_status = "fallback"`
10. 问股成功返回前写入 `records` 表。
11. 问股记录写入条件：
   - 行情和技术指标成功；
   - 无论 AI 成功还是规则回退，都算问股成功；
   - 只有行情失败、输入错误时不写记录。
12. 问股记录标题：
   - `问股：{stock_name}（{stock_code}）`
13. 问股记录 summary：
   - 优先取回答前 80 字；
   - 或使用 `{stock_name} 当前价 {price}，评分 {score}，建议 {action}`。
14. 问股记录 `question` 保存用户原始问题。
15. 问股记录 `answer` 保存最终展示给用户的回答。
16. `metadata_json` 保存价格、涨跌幅、评分、建议、趋势、模型、AI 状态。

### 5.5 analysis API 改造

- `backend/app/api/analysis.py`

7. 分析记录 summary：
   - 使用报告 summary。
8. 分析记录 `report_id` 关联新生成的报告。
9. `metadata_json` 保存：
   - price
   - change_pct
   - score
   - action
   - trend
   - task_id
10. `GET /api/analysis/{task_id}` 增加默认用户过滤。

## 8. llm_error 处理计划

当前后端在 `backend/app/api/ask.py` 的 `AskResponse` 暴露 `llm_error`。

v0.4 计划：

1. 不再向前端返回 `llm_error`。
2. 后端内部捕获 `LlmError` 后只影响：
   - `answer_type = "rule"`
   - `ai_status = "fallback"`
3. 前端不再定义 `llm_error` 字段。
4. 前端不展示底层 LLM 错误。
5. 用户看到的是稳定产品文案，而不是开发调试信息。
