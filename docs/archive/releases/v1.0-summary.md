# v1.0 前端架构升级（Zustand 状态管理）总结

## 1. 版本定位

v1.0 的目标是替换 v0.9 引入的 DataRefreshContext 版本号信号模式，用 Zustand 真正的共享状态管理替代"版本号 + useEffect 检测"的补丁方案。

本版本不改变 UI、不新增功能、不改后端，只做一件事：

```text
把状态从页面 useState + Context 信号模式，迁移到 Zustand Store
```

## 2. 已完成内容

### 2.1 安装 Zustand

- 新增依赖：`zustand`
- 文件：`mobile-app/package.json`

### 2.2 类型拆分

原来所有类型集中在 `types/index.ts` 一个文件，按领域拆分为 5 个文件：

```
types/
  index.ts        → 只做 re-export
  stock.ts        → Stock, AnalysisTask

  ask.ts          → AskResponse, AskMessage
  record.ts       → RecordItem, RecordDetail
```

### 2.3 创建 stores 基础结构

新增目录 `mobile-app/src/stores/`：

```
stores/
  index.ts            → 统一导出 + resetAllStores()
  watchlistStore.ts   → 自选股 + 任务状态
  recordsStore.ts     → 问股记录
  askStore.ts         → 问股会话
```

### 2.4 watchlistStore

状态：
- `stocks: Stock[]` — 自选股列表
- `taskStatuses: Record<string, string>` — 各股票的分析任务状态
- `isLoading: boolean` — 加载中
- `loadError: string` — 加载错误

操作：
- `loadStocks()` — 调用 `apiGet('/api/stocks')`
- `loadTaskStatuses()` — 从 AsyncStorage 读取 taskIds 并逐个查询状态
- `addStock(code, name)` — 添加自选，成功后自动重新加载列表
- `deleteStock(code)` — 删除自选，清理 AsyncStorage，自动重新加载列表
- `createAnalysis(stockCode)` — 创建分析任务，保存 taskId
- `updateTaskStatus(code, status)` — 供 TaskStatusScreen 回调更新

### 2.5 recordsStore

状态：
- `items: RecordItem[]` — 记录列表
- `isLoading: boolean` — 加载中
- `loadError: string` — 加载错误

操作：
- `fetchRecords()` — 调用 `apiGet('/api/records')`

### 2.6 askStore

状态：
- `sessionId: string | null` — 当前会话 ID
- `messages: AskMessage[]` — 消息列表
- `question: string` — 输入框内容
- `isLoading: boolean` — 加载中
- `error: string` — 错误信息
- `latestResult: AskResponse | null` — 最新 AI 回答

操作：
- `setQuestion(text)` — 设置输入框内容
- `handleAsk()` — 发送问题，调用 `apiPost('/api/ask', ...)`
- `handleNewSession()` — 重置会话
- `handleAddToWatchlist()` — 加入自选，成功后联动 `watchlistStore.getState().loadStocks()`
- `restoreSession(sessionId, messages)` — 从记录详情恢复会话

### 2.7 迁移 WatchlistScreen

- 移除 `useState` 的 stocks/taskStatuses/isLoading/loadError
- 移除 `useDataRefresh()` 引用
- 移除 `TASK_IDS_KEY` / `readTaskIds` / `saveTaskIds` 本地 AsyncStorage 逻辑（迁移到 store 内部）
- 改为 `useWatchlistStore()` 获取状态和操作

### 2.8 迁移 RecordListScreen

- 移除 `useState` 的 items/loading/error
- 移除 `useDataRefresh()` 引用
- 改为 `useRecordsStore()` 获取状态和操作

### 2.9 迁移 AskScreen

- 移除 `useState` 的 sessionId/messages/question/loading/error/latestResult/addToWatchlistLoading
- 移除 `useDataRefresh()` 引用
- 改为 `useAskStore()` 获取状态和操作
- 保留 `initialParamsHandled` 逻辑（导航参数是页面级关注点）

### 2.10 迁移 TaskStatusScreen

- 移除 `useDataRefresh()` 引用
- 任务完成时直接调用 `watchlistStore.getState().loadStocks()` / `loadTaskStatuses()` / `recordsStore.getState().fetchRecords()`
- 保留 3 秒轮询逻辑

### 2.11 更新 MineScreen

- 移除 `useDataRefresh()` 引用
- 登出时调用 `resetAllStores()` 清空所有 store 数据

### 2.12 更新 LoginScreen

- 移除 `useDataRefresh()` 引用
- 登录成功后直接 `navigation.goBack()`，页面重新渲染时自然重新 fetch

### 2.13 更新 useApiErrorHandler

- 移除 `useDataRefresh()` 依赖
- 401 时调用 `resetAllStores()` 替代 `notifyAllDataChanged()`

### 2.14 API 层增强

- 文件：`mobile-app/src/api/client.ts`
- 401 时自动清除 token（`AsyncStorage.removeItem(STORAGE_KEYS.AUTH_TOKEN)`）

### 2.15 删除 DataRefreshContext

- 删除文件：`mobile-app/src/contexts/DataRefreshContext.tsx`

## 6. 学习重点

本版本重点学习了：

1. **Zustand 核心概念**：`create()`、`set()`、`get()`、`getState()`
2. **状态管理分层原则**：useState（局部 UI）→ AuthContext（低频全局）→ Zustand（共享业务数据）
3. **Store 间联动模式**：`StoreA.getState().actionInStoreB()` 替代版本号信号
4. **resetAllStores 模式**：登出/401 时一键清空所有 store
5. **数据流变化**：从"操作 → notify → 版本号 +1 → useEffect 检测 → 重新 fetch"到"操作 → 直接调用目标 store action"
6. **计划质量门禁**：涉及删除/替换公共模块时，必须先 Grep 全量引用再出计划
