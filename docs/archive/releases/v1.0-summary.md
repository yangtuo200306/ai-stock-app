# v1.0 阶段一架构升级总结

## 1. 版本定位

v1.0 的目标是完成阶段一"全局架构升级"，分两个子阶段推进：

```text
1A：前端架构升级 — Zustand 状态管理替代 DataRefreshContext 版本号信号模式
1B：后端架构升级 — 配置管理、日志系统、错误码体系、数据库迁移
```

本版本不改变 UI、不新增功能，只改骨架。

---

## 2. 子阶段 1A：前端架构升级

### 2.1 安装 Zustand

- 新增依赖：`zustand`
- 文件：`mobile-app/package.json`

### 2.2 类型拆分

原来所有类型集中在 `types/index.ts` 一个文件，按领域拆分为 5 个文件：

```
types/
  index.ts        → 只做 re-export
  stock.ts        → Stock, AnalysisTask
  report.ts       → Report, ReportIndicators, ReportHistoryItem
  ask.ts          → AskResponse, AskMessage
  record.ts       → RecordItem, RecordDetail
  navigation.ts   → 所有导航参数类型
```

### 2.3 创建 stores 基础结构

新增目录 `mobile-app/src/stores/`：

```
stores/
  index.ts            → 统一导出 + resetAllStores()
  watchlistStore.ts   → 自选股 + 任务状态
  recordsStore.ts     → 问股记录
  askStore.ts         → 问股会话
```

### 2.4 迁移页面

- **WatchlistScreen** — 移除 useState 的 stocks/taskStatuses/isLoading/loadError，移除 useDataRefresh()
- **RecordListScreen** — 移除 useState 的 items/loading/error，移除 useDataRefresh()
- **AskScreen** — 移除 useState 的 sessionId/messages/question/loading/error/latestResult，移除 useDataRefresh()
- **TaskStatusScreen** — 移除 useDataRefresh()，任务完成时联动调用 store action
- **MineScreen** — 移除 useDataRefresh()，登出时调用 resetAllStores()
- **LoginScreen** — 移除 useDataRefresh()，登录成功后直接 goBack()

### 2.5 API 层增强

- 401 时自动清除 token（`AsyncStorage.removeItem(STORAGE_KEYS.AUTH_TOKEN)`）

### 2.6 删除 DataRefreshContext

- 删除文件：`mobile-app/src/contexts/DataRefreshContext.tsx`
- 更新：`mobile-app/src/navigation/AppNavigator.tsx` — 移除 `DataRefreshProvider`

### 2.7 版本号更新

- MineScreen：`当前版本：v0.8` → `当前版本：v1.0`

## 3. 子阶段 1B：后端架构升级

### 3.1 配置管理

- 新增 `backend/app/config/settings.py`，使用 `pydantic-settings` 集中管理 10 项配置
- 所有 `os.environ.get` 调用替换为 `settings.xxx`
- 移除 `llm_client.py` 中的手动 `load_dotenv()` 调用
- CORS origins 通过 `cors_origins_list` property 解析逗号分隔字符串

### 3.2 日志系统

- 新增 `backend/app/logging_config.py`，三级输出：
  - 控制台（按 LOG_LEVEL）
  - 常规日志文件（INFO，10MB 轮转，保留 5 个备份）
  - 调试日志文件（DEBUG，50MB 轮转，保留 3 个备份）
- 12 个文件添加 `import logging` + `logger = logging.getLogger(__name__)` + 关键路径日志
- 第三方库降噪（urllib3、httpx → WARNING）
- health.py 不加日志（高频调用）

### 3.3 错误码体系

- 新增 `backend/app/errors.py`，定义 13 个 `ErrorCode` 枚举 + `error_response()` / `api_error()` 辅助函数
- 新增 `backend/app/error_handler.py`，全局异常处理器兜底
- 替换 17 处散落的 error_code 字符串为枚举引用
- 修复 `analysis.py` 重复 `"message"` key bug（后面的覆盖前面的）
- 统一 `analysis.py` 行情失败处理风格（返回 dict → 抛 HTTPException）

### 3.4 数据库迁移

- 新增 `schema_version` 表 + 版本号常量
- 3 个版本化迁移：
  - V20260625_001_initial_baseline：初始建表
  - V20260625_002_migrate_stocks_unique：唯一约束迁移
  - V20260625_003_ensure_columns：补充缺失列
- 启动时按版本顺序执行未应用的迁移

## 7. 学习重点

### 1A 学习重点

1. **Zustand 核心概念**：`create()`、`set()`、`get()`、`getState()`
2. **状态管理分层原则**：useState（局部 UI）→ AuthContext（低频全局）→ Zustand（共享业务数据）
3. **Store 间联动模式**：`StoreA.getState().actionInStoreB()` 替代版本号信号
4. **resetAllStores 模式**：登出/401 时一键清空所有 store
5. **数据流变化**：从"操作 → notify → 版本号 +1 → useEffect 检测 → 重新 fetch"到"操作 → 直接调用目标 store action"
6. **计划质量门禁**：涉及删除/替换公共模块时，必须先 Grep 全量引用再出计划

### 1B 学习重点

1. **pydantic-settings**：自动读 .env + 类型转换 + 校验，替代手动 `os.environ.get` + `load_dotenv()`
2. **日志系统设计**：三级输出 + RotatingFileHandler + 第三方库降噪
3. **错误码枚举**：用 `ErrorCode` 枚举替代散落字符串，改一处全生效
4. **全局异常处理器**：兜底未预期异常，保证统一 JSON 格式
5. **版本化迁移**：schema_version 表 + 有序迁移列表，增量演进
