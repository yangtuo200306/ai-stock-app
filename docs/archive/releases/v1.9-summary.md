## 各版本详细总结

### v1.9.1：Prompt 工程优化

**核心改动：**
- System Prompt 从"你是股票分析助手"升级为结构化角色定义 + 回答规范
- 统一输出格式：结论→关键数据→详细分析→风险提示
- 指标按趋势/动量/成交量分类，加括号说明文字

**完成后效果：**
- AI 回答结构清晰，先给结论再展开分析
- 关键数据加粗展示，用户扫一眼就能抓住重点
- 指标分类呈现，小白也能看懂

### v1.9.2：流式响应

**核心改动：**
- 后端新增 `POST /api/ask/stream` SSE 流式端点
- 前端 `apiPostStream()` 逐 chunk 渲染消息，显示光标闪烁
- 保留非流式降级路径

**完成后效果：**
- 回答逐字出现，用户不用干等 3-5 秒
- 为 Agent（多次调工具更慢）做好体验铺垫

### v1.9.3：资讯/新闻集成

**核心改动：**
- 新增 `news_fetcher.py` 独立模块，akshare 东方财富（主源）+ 搜狗（备源）双源获取
- 问股时自动检索相关新闻，塞入 Prompt，LLM 在回答中引用
- 前端新增 NewsCard 组件，MessageBubble 底部展示新闻来源卡片
- 分析报告底部新增"相关资讯"区块
- 30 分钟缓存 TTL，三级降级

**完成后效果：**
- AI 回答有新闻依据，不再只有技术面分析
- 问股结果附带相关新闻条目，可点击跳转原文

### v1.9.4：轻量 Agent（Function Calling）

**核心改动：**
- 新增 `tool_registry.py` + `tool_factory.py` — 工具注册表模式
- 新增 `agent_loop.py` — ReAct 循环引擎（max_steps=5, 超时保护）
- 新增 `POST /api/ask/agent/stream` 端点
- 前端新增可折叠思考过程面板（thinking → tool_start → tool_done → text）
- 右上角 Agent/标准模式切换按钮
- 三层降级保护：Agent → 传统流式 → 规则回答

**修复清单（8 个问题）：**
- `get_stock_quote` 字段不匹配导致报错
- 流式输出改为逐 chunk 透传
- 记录保存改用 `json.loads()` 正确解析
- 排除 `search_stock` 减少一轮 LLM 调用
- 优化思考过程事件序列
- system prompt 加结构化格式要求
- 新闻用 Markdown 链接输出
- 内容溢出加 CSS 保护

**完成后效果：**
- AI 能主动查数据，不再被动等喂数据
- 用户能看到 AI 思考→查数据→分析的全过程
- 两种模式可对比测试

### v1.9.5：Agent 模式基础优化

**核心改动：**
1. **输入零限制** — 去掉 `stock_code` 强制校验，LLM 自行通过 `search_stock` 搜索
2. **原则式 Prompt** — 告诉 LLM"不能做什么"而非"怎么做"，自主决定回答结构
3. **加回 `search_stock`** — 工具选择权还给 LLM
4. **多轮 context 通道** — system 角色注入上一轮股票上下文和问题
5. **思考过程持久化** — `ask_messages` 表新增 `thinking_json` 列，历史消息可回溯

**附加优化：**
- 无 stock_code 时跳过 `_build_minimal_report`，避免无用日志
- 通用会话标题用用户问题前 20 字
- 空状态提示文字随模式切换
- 降级路径兼容无 stock_code 场景

**完成后效果：**
- 用户可输入"茅台怎么样"、"今天大盘如何"等自然语言
- AI 回答结构灵活，根据问题自主决定组织形式
- 多轮对话上下文连贯
- 历史消息的思考过程可回溯查看

## 关键设计决策

### 决策 1：Agent 模式不动标准模式

Agent 模式是独立的新端点 `POST /api/ask/agent/stream`，标准模式的 `POST /api/ask` 和 `POST /api/ask/stream` 完全不受影响。两种模式通过右上角按钮切换。

### 决策 2：Tool Registry 模式

```python
class ToolRegistry:
    _tools: dict[str, ToolDefinition]
    def register(self, tool): ...
    def get_openai_tools(self): ...
    def execute(self, name, **kwargs): ...
```

加新能力 = 注册一个新 tool，零管线改动。

### 决策 3：SSE 多类型事件协议

Agent 循环是多步的，用 SSE 逐条推送不同类型的事件：

```
data: {"type":"thinking","message":"AI 正在分析..."}
data: {"type":"tool_start","tool":"get_stock_quote","display_name":"获取实时行情"}
data: {"type":"tool_done","tool":"get_stock_quote","success":true,"duration":0.5}
data: {"type":"text","content":"根据最新数据..."}
data: {"type":"done","success":true,"session_id":"xxx"}
```

### 决策 4：原则式 Prompt

v1.9.5 将 Agent 的 system prompt 从模板式改为原则式。模板式适合确定性输出（标准模式），原则式适合非确定性编排（Agent 模式）。

### 决策 5：JSON 列持久化思考过程

思考过程的特点是"写入一次，读取展示"，用 `ask_messages` 表的 `thinking_json` 可空 TEXT 列存储。新列可空，旧数据兼容。

## 踩坑记录

### 1. JSON 字符串匹配不靠谱（v1.9.4）

`json.dumps` 输出 `"type": "text"`（冒号后有空格），但手写检查条件 `'"type":"text"'`（无空格）永远匹配不上。必须用 `json.loads()` 正确解析。

### 2. Agent 流式输出不是真流式（v1.9.4）

LLM 文本累加后一次性 yield，前端看到的是整段文字突然出现。修复：改为逐 chunk 透传。

### 3. 竞态问题：降级回调（v1.9.4）

`done` 事件回调里直接调降级函数，会和 `onDone` 回调形成竞态。修复：用 `shouldFallback` 标记延迟到 `onDone` 再执行。

### 4. 降级路径兼容性（v1.9.5）

Agent 降级到标准模式时，如果用户没提供 stock_code，标准模式仍然会报 400。修复：在降级前判断是否有 stock_code。

### 5. 数据链路透传遗漏（v1.9.3）

`build_analysis_report` 自己拼了 `indicators` dict，漏了 `turnover_rate` 和 `amplitude`。虽然 `technical_indicators.py` 返回了、`market_data.py` 提取了，但中间环节没接上。
