## 核心改动详解

### 1. 数据结构标准化

改前：indicators 是松散 dict，前后端类型靠人工对齐

改后：
- 后端：`Indicators` Pydantic 模型（17 字段，全部可选）
- 前端：`ReportIndicators` TypeScript 接口（与后端一一对应）
- FastAPI 自动校验，OpenAPI 文档自动生成

### 2. 数据展示优化

**自选卡片：**
```
改前：代码/名称/价格/涨跌幅/摘要
改后：代码/名称/价格/涨跌幅 + 评分徽标/趋势标签/建议标签 + 摘要
```

**问股结果 & 记录详情 & 分析报告：**
```
改前：行情概览 + 技术指标（平铺）
改后：行情概览（评分/趋势/建议/涨跌幅/换手率/振幅/成交量）
      技术指标（MA5/10/20/均线趋势/RSI(6)/乖离率）
```

### 3. 指标调整

| 指标 | 操作 | 理由 |
|------|------|------|
| 换手率 | 新增 | efinance 有，反映活跃度 |
| 振幅 | 新增 | 用(最高-最低)/昨收×100 计算 |
| RSI(12) | 去掉 | 和 RSI(6) 重复，对小白是干扰 |
| 指标标签 | 加括号解释 | 如 MA5（5日均线）、RSI(6)（相对强弱指标） |

### 4. 顺手清理

- ScoreGauge.tsx 内部重复的 getScoreColor → 改用公共函数
- AskScreen.tsx 加 `?.` 安全访问
- AskScreen.tsx 的 resultCard 改用 AppCard 组件

## 设计决策

### 决策 1：记录详情技术指标来源

- **方案：** 写记录时把 indicators 写入 metadata_json
- **理由：** 一次查询搞定，不改表结构
- **影响：** ask_service.py 和 analysis.py 的 metadata 构建逻辑

### 决策 2：新浪备用源字段处理

- **策略：** 能拿的拿（成交量、成交额、振幅计算），拿不到的留 null
- **前端处理：** MetricCard 对 null 值显示 `--`

### 决策 3：评分原因展示

- **位置：** ReportDetailScreen 的摘要后面，独立成块
- **理由：** 放在评分下面太挤，放在行情概览外面更清晰

## 踩坑记录

**问题：** `build_analysis_report` 自己拼了 `indicators` dict，漏了 `turnover_rate` 和 `amplitude`。虽然 `technical_indicators.py` 返回了、`market_data.py` 提取了，但中间环节没接上，导致 API 返回始终为 None。

**教训：** 改数据链路时，要检查每个环节是否都透传了新字段。`docs/guide/data-flow.md` 里有完整的链路图，改之前应该先看。
