# v0.9 状态管理与稳定性最小版总结

## 1. 版本定位

v0.9 的目标是解决 v0.8 后最影响体验的状态问题：登录切换、token 失效、页面间数据同步、任务状态刷新。

本版本不引入 Zustand / React Query 等复杂状态库，也不做完整缓存体系，只做三件事：

1. 登录 / 登出 / 切换账号后，业务状态稳定刷新；
2. token 运行中失效时，统一清理并提示重新登录；
3. 问股、自选、记录、任务之间有轻量同步机制。

本版本依然延续 v0.8 已定型的 `自选 | 问股 | 记录 | 我的` 四个 Tab，不新增大功能，不调整后端接口契约。

新增目录：

实现内容：

1. `watchlistVersion`：自选相关状态版本号。
2. `recordsVersion`：记录相关状态版本号。
3. `tasksVersion`：任务相关状态版本号。
4. `notifyWatchlistChanged()`：触发自选刷新。
5. `notifyRecordsChanged()`：触发记录刷新。
6. `notifyTasksChanged()`：触发任务状态刷新。
7. `notifyAllDataChanged()`：登录 / 登出 / 切换账号时全部刷新。

设计原则：

- 只负责通知刷新，不持有真实业务数据；
- 不做缓存，不做乐观更新；
- 页面各自重新请求后端，保持当前页面状态管理模式。

### 2.5 新增 401 / token 失效统一处理

新增目录：

```text
mobile-app/src/hooks/
```

新增文件：

`useApiErrorHandler.ts`

完成内容：

1. 统一识别 `status === 401` 或 `errorCode` 为 token 失效；
2. 失效时自动调用 `clearSession()` 清内存状态；
3. 失效时自动调用 `notifyAllDataChanged()` 触发所有页面刷新；
4. 返回统一提示文案：`"登录状态已失效，请重新登录"`；
5. 非失效错误返回原始错误文案或 fallback message；
6. 返回 `isAuthExpired` 标记，页面可据此做跳转或其他处理。

## 5. 本版本不做内容

v0.9 明确不做：

1. 不引入 Zustand / Redux / MobX。
2. 不引入 React Query / SWR。
3. 不做完整状态缓存。
4. 不做 WebSocket / SSE 实时推送。
5. 不重构后端异步任务。
6. 不引入测试。
7. 不做部署。
8. 不 push。

## 6. 学习重点

本版本重点学习了：

1. 刷新信号（版本号） vs 状态缓存（Zustand）的取舍：先做边界，再做缓存。
2. `ApiError` 类不是为了少写代码，而是为了让错误类型可断言、可统一判断、可在页面间复用。
3. `useApiErrorHandler` 封装了 401 后的公共动作：清 session + 触发刷新 + 统文案。
4. `setInterval` 清理很容易遗漏，必须用 ref 保存并在 effect 清理函数中清除。
5. 先实现最小稳定，再做状态升级。v0.9 做刷新信号层，后续再过渡到真正的状态缓存层。
6. 不要为了消除 any 而重构整个导航结构，版本内只做低风险收口。
7. `notifyAllDataChanged` 让登录 / 登出 / 切换账号的页面数据刷新不需要硬编码每个页面。
