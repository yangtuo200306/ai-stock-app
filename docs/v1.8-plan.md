# v1.8 计划 — 股票数据体系优化

> 阶段四：让数据更清晰、更完整、更可靠

---

## 一、为什么做这个阶段？

项目经过三个阶段后，功能已完整、服务已上线，但股票数据的展示和流转存在以下问题：

### 当前问题清单

| # | 问题 | 影响 | 涉及范围 |
|---|------|------|---------|
| 1 | 自选列表只显示名称和价格，看不到评分/趋势/建议 | 用户需要点进去才知道股票状况 | 前端 WatchlistScreen |
| 2 | 技术指标（MA、RSI、成交量）散落在文字回答里，没有结构化展示 | 用户难以快速获取关键指标 | 前端 AskScreen / RecordDetailScreen |
| 3 | 后端 indicators 字段是松散 dict，前后端类型靠人工约定 | 改字段容易漏改前端 | 后端 report_builder.py / 前端 types |
| 4 | 问股返回的数据和自选分析返回的数据字段不一致 | 前端需要兼容两种格式 | 后端 ask.py / analysis.py |
| 5 | 评分展示只有水平进度条，不够直观 | 视觉体验一般 | 前端 ScoreGauge |
| 6 | 记录详情不展示技术指标明细 | 历史记录信息量不足 | 前端 RecordDetailScreen |
| 7 | 行情数据只有价格和涨跌幅，缺少换手率、量比、振幅等 | 信息维度单一 | 后端 market_data.py |
| 8 | 数据源只有 efinance + 新浪，没有其他备用 | 单一数据源风险 | 后端 market_data.py |

### 参考项目可借鉴的模式

参考项目 `stock-analysis-reference` 在数据展示方面有以下值得借鉴的设计：

| 借鉴点 | 参考项目做法 | 手机端适配方式 |
|--------|------------|--------------|
| 报告概览布局 | 两列布局展示名称/价格/涨跌幅/趋势/评分 | 改为上下流式卡片 |
| 评分仪表盘 | 270 度弧形 SVG，颜色随分数变化 | 用 react-native-svg 移植 |
| 自选股列表 | 每条显示名称+评分+建议+分析次数 | 增强当前 WatchlistStockCard |
| 指标卡片 | StatCard：标签+数值+色调 | 新建 MetricCard 组件 |
| 决策信号 | 结构化展示买入/卖出信号 | 可后续考虑 |
| 统一行情结构 | UnifiedRealtimeQuote 定义完整字段 | 作为类型定义参考 |

## 二、目标

**让股票数据更好看、更完整、更可靠。**

具体来说：

1. **统一数据** — 前后端数据结构对齐，类型安全，问股和自选分析返回一致
2. **优化展示** — 每个页面展示最相关的数据，结构化、卡片化、直观化
3. **完善指标** — 补充有价值的指标（如换手率、量比），去除冗余
4. **评估数据源** — 看看是否需要加腾讯数据源等备用源

---

## 三、执行步骤（共 3 步）

### 第 1 步：统一数据结构与类型定义

**目标：** 前后端数据字段对齐，类型安全，为展示优化打好基础。

**涉及文件：**

```
后端：
  backend/app/services/report_builder.py  — 标准化 indicators 字段
  backend/app/api/ask.py                  — 对齐 AskResponse 字段
  backend/app/api/analysis.py             — 对齐分析返回字段
  backend/app/api/stocks.py               — 自选列表返回增强

前端：
  mobile-app/src/types/report.ts          — ReportIndicators 完整定义
  mobile-app/src/types/stock.ts           — Stock 类型增强
  mobile-app/src/types/ask.ts             — AskResponse 对齐
  mobile-app/src/types/record.ts          — RecordItem/Detail metadata 对齐
```

**具体任务：**

1.1 梳理当前后端返回的所有股票数据字段，列出完整清单
1.2 定义标准化的 `Indicators` 数据结构（后端 Pydantic + 前端 TypeScript）
1.3 确保 ask.py 和 analysis.py 返回的 indicators 字段结构一致
1.4 自选列表接口返回评分、趋势、操作建议
1.5 记录详情接口返回完整技术指标

**验收标准：**
- 前后端类型定义一一对应，无松散字段
- 问股和自选分析返回的 indicators 结构一致
- 自选列表每条显示评分和趋势
- 记录详情展示技术指标

---

### 第 2 步：优化数据展示

**目标：** 每个页面展示最相关的数据，结构化、卡片化、直观化。

**涉及文件：**

```
前端：
  mobile-app/src/screens/WatchlistScreen.tsx     — 自选列表增强
  mobile-app/src/screens/AskScreen.tsx           — 问股结果结构化展示
  mobile-app/src/screens/RecordDetailScreen.tsx  — 记录详情展示指标
  mobile-app/src/components/WatchlistStockCard.tsx — 自选卡片增强
  mobile-app/src/components/ScoreGauge.tsx        — 评分展示优化
  mobile-app/src/components/                       — 新增 MetricCard 等组件
  mobile-app/src/utils/stockDisplay.ts            — 展示工具函数
```

**具体任务：**

2.1 **自选列表增强**
  - 每条显示评分（颜色标识）、趋势标签、操作建议
  - 价格和涨跌幅展示优化（字体、颜色）
  - 卡片布局微调，信息层级更清晰

2.2 **评分展示优化**
  - 当前：水平进度条
  - 目标：颜色区分更明显，分数更突出
  - 参考：参考项目的 ScoreGauge（SVG 弧形），但不强求，先优化颜色和布局

2.3 **问股结果结构化展示**
  - 当前：文字回答 + 评分条
  - 目标：在回答上方增加指标摘要卡片（MA 趋势、RSI、成交量）
  - 参考：参考项目的 StatCard 和 ReportOverview

2.4 **记录详情展示指标**
  - 当前：只显示文字回答
  - 目标：在回答上方展示当时的技术指标摘要

2.5 **新建 MetricCard 组件**
  - 通用指标卡片：标签 + 数值 + 色调
  - 用于展示 MA、RSI、成交量等指标
  - 参考：参考项目的 StatCard

**验收标准：**
- 自选列表一眼能看出每只股票的评分和趋势
- 问股结果上方有指标摘要卡片
- 记录详情能看到历史技术指标
- 评分展示颜色直观

---

### 第 3 步：完善指标与评估数据源

**目标：** 补充有价值的指标，评估是否需要加新数据源。

**涉及文件：**

```
后端：
  backend/app/services/technical_indicators.py  — 新增指标计算
  backend/app/services/market_data.py           — 新增行情字段
  backend/app/services/report_builder.py        — 新指标加入报告

前端：
  mobile-app/src/types/report.ts                — 新指标类型
  mobile-app/src/components/                     — 新指标展示组件
```

**具体任务：**

3.1 **梳理当前指标**
  - 当前已有：MA5/10/20、乖离率、均线趋势、RSI(6/12)、成交量比值/信号
  - 评估哪些有用、哪些冗余
  - 参考项目的 UnifiedRealtimeQuote 字段清单

3.2 **补充有价值的指标**
  - 考虑增加：换手率、振幅、量比（已有 volume_ratio）
  - 考虑增加：MACD（金叉/死叉信号）
  - 考虑增加：支撑位/压力位（简单版本）

3.3 **评估数据源**
  - 当前：efinance（主）+ 新浪（备用）
  - 参考项目有 10 个数据源
  - 评估腾讯数据源（稳定性好、接口简单）
  - 决策：是否加新数据源，加哪个

**验收标准：**
- 指标清单清晰，无冗余
- 新增指标计算正确
- 数据源评估结论明确（加/不加）

---

## 四、不做范围

- 不改 AI 模型调用（Prompt、模型切换等）
- 不加新功能（持仓、回测、多市场等）
- 不改数据库表结构（JSON 字段已足够扩展）
- 不加 K 线图（参考项目也没有，且实现成本高）
- 不改导航结构
- 不改认证/用户系统

## 六、数据流变化

### 当前数据流

```
行情源 → market_data.py → quote + history
  → technical_indicators.py → indicators dict
  → report_builder.py → report dict（含 indicators）
  → ask.py / analysis.py → 返回给前端
  → 前端展示（文字为主）
```

### 目标数据流

```
行情源 → market_data.py → quote + history（新增字段）
  → technical_indicators.py → indicators（标准化结构，新增指标）
  → report_builder.py → report（标准化字段）
  → ask.py / analysis.py → 统一返回结构
  → 前端展示（卡片化、结构化）
       ├── 自选列表：名称 + 价格 + 涨跌 + 评分 + 趋势
       ├── 问股结果：指标摘要卡片 + AI 回答
       ├── 记录详情：历史指标 + 历史回答
       └── 报告详情：完整指标展示
```
