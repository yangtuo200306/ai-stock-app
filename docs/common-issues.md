# 常见问题与排查手册

这个文件用于记录项目开发过程中容易反复遇到的问题。

记录重点：

- 问题现象；
- 常见原因；
- 新手解释；
- 解决方式；
- 以后怎么避免。

## 1. 前端页面不刷新

### 现象

修改 `mobile-app/App.tsx` 后，浏览器页面没有变化。

### 原因

常见原因包括：

- Expo / Metro 没有自动刷新；
- 浏览器缓存没有更新；
- 开发服务处于特殊环境变量影响下，例如 `CI=true`；
- 页面打开的不是当前 Expo Web 服务地址。

### 新手解释

可以理解为：

```text
代码已经改了
但浏览器还在看旧画面
```

前端开发服务需要重新打包，浏览器也需要重新加载新代码。

### 解决方式

先尝试刷新浏览器：

```text
Ctrl + F5
```

如果还不行，重新启动 Expo Web：

```text
npm run web -- --clear
```

如果终端提示 `reloads are disabled`，可以检查是否存在 `CI=true` 的影响。

### 以后怎么避免

- 修改页面后先看终端是否重新构建；
- 页面没变时先强制刷新浏览器；
- 遇到缓存问题再用 `--clear`；
- 确认打开的是 Expo 输出的正确地址。

## 2. 后端改代码后不生效

### 现象

修改后端代码后，例如修改 `backend/app/main.py`，前端请求结果还是和修改前一样。

### 原因

后端服务启动后，运行的是启动时加载到内存里的旧代码。

如果启动时没有加 `--reload`，修改 `.py` 文件后服务不会自动重载。

### 新手解释

可以理解为：

```text
代码文件 = 菜谱
运行中的后端服务 = 厨师正在按旧菜谱做菜
```

改了菜谱以后，厨师不一定自动换新菜谱。

### 解决方式

开发阶段推荐用：

```text
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000 --reload
```

这样修改 Python 文件后，Uvicorn 会自动重启服务。

### 以后怎么避免

- 开发阶段启动后端时加 `--reload`；
- 修改后端入口、路由、数据库初始化等代码后，确认服务已重载；
- 看终端是否出现重新加载或启动完成的日志。

## 3. 前端请求后端失败：net::ERR_FAILED

### 现象

前端点击“测试连接”后失败，浏览器控制台出现：

```text
net::ERR_FAILED http://127.0.0.1:8000/api/health
```

### 原因

常见原因包括：

- 后端服务没有启动；
- 后端地址输入错误；
- FastAPI 没有配置 CORS；
- 前端请求地址和后端实际地址不一致。

本项目中遇到的主要原因是：

```text
前端 Web 和后端 API 跨域
```

前端 Web 地址：

```text
http://localhost:8081
```

后端地址：

```text
http://127.0.0.1:8000
```

浏览器认为它们是不同来源。

### 新手解释

可以理解为：

```text
前端是小区 A 的人
后端是小区 B 的门卫
```

浏览器会检查：

```text
后端门卫有没有允许这个前端进来
```

如果后端没有配置 CORS，浏览器就会拦截请求。

### 解决方式

在 FastAPI 入口 `backend/app/main.py` 中统一配置 CORS。

开发阶段允许：

```text
http://localhost:8081
http://127.0.0.1:8081
```

配置后需要重启后端服务，或者使用 `--reload` 自动重载。

### 以后怎么避免

- 先直接访问 `http://127.0.0.1:8000/api/health` 确认后端可用；
- 前端输入框只填基础地址，例如 `http://127.0.0.1:8000`；
- 不要把 `/api/health` 也填进输入框，因为代码会自动拼接；
- Web 前端请求后端时，优先检查 CORS。

## 4. TSX 代码区裸写 URL 报错

### 现象

IDE 报类似错误：

```text
此处不允许使用 'A 标签
```

或 TSX 解析异常。

### 原因

把 URL 直接写进了 TypeScript / TSX 代码区，例如：

```text
http://127.0.0.1:8000
```

如果它不在字符串、注释或 JSX 文本中，编译器会把它当成错误代码解析。

### 新手解释

可以理解为：

```text
代码区只能放合法代码
不能随手写一行普通地址
```

### 解决方式

如果要保存 URL，写成字符串：

```ts
const url = 'http://127.0.0.1:8000';
```

如果要显示 URL，放进 `Text`：

```tsx
<Text>http://127.0.0.1:8000</Text>
```

如果要作为输入框提示，放进 `placeholder`：

```tsx
placeholder="http://127.0.0.1:8000"
```

### 以后怎么避免

- 不要在函数中间裸写 URL；
- 看到奇怪 TSX 报错时，先检查附近是否误输入普通文字；
- URL 要么放字符串，要么放组件文本，要么放注释。

## 5. placeholder 不是输入值

### 现象

页面刚打开，用户没有输入，但输入框里看起来已经有地址。

例如：

```text
http://127.0.0.1:8000
```

### 原因

这是 `placeholder` 提示文字，不是真正的输入值。

真实输入值由 `value={backendUrl}` 决定。

### 新手解释

可以理解为：

```text
placeholder = 作业本上的浅色示例答案
backendUrl = 用户真正写上去的答案
```

浅色示例看起来在输入框里，但它不是用户输入的内容。

### 解决方式

通过页面上的“当前输入”判断真实状态。

如果显示：

```text
当前输入：暂未输入
```

说明 `backendUrl` 其实是空的。

如果 placeholder 颜色太深，可以设置：

```tsx
placeholderTextColor="#94a3b8"
```

### 以后怎么避免

- 不要把 placeholder 当成真实值；
- 需要默认值时，应设置 state 初始值；
- 需要提示示例时，用 placeholder。

## 6. 127.0.0.1 在电脑和手机上的含义不同

### 现象

Web 预览能访问后端，但以后真机 APP 可能访问失败。

### 原因

`127.0.0.1` 表示“本机”。

在电脑浏览器里：

```text
127.0.0.1 = 电脑自己
```

在手机真机里：

```text
127.0.0.1 = 手机自己
```

所以手机访问 `127.0.0.1:8000` 时，不是在访问电脑后端。

### 新手解释

可以理解为：

```text
每台设备说“我自己”时，指的都是自己，不是别人
```

电脑说 `127.0.0.1` 是电脑自己。

手机说 `127.0.0.1` 是手机自己。

### 解决方式

Web 预览阶段可以使用：

```text
http://127.0.0.1:8000
```

以后真机访问电脑后端时，需要使用电脑局域网 IP，例如：

```text
http://192.168.x.x:8000
```

同时后端启动时可能需要监听：

```text
0.0.0.0
```

而不是只监听：

```text
127.0.0.1
```

### 以后怎么避免

- Web 预览和真机调试要区分地址；
- 真机访问电脑后端时，用电脑局域网 IP；
- 确认手机和电脑在同一个局域网；
- 确认防火墙没有拦截端口。

## 7. React Native Web：Unexpected text node

### 现象

Expo Web 控制台报错：

```text
Unexpected text node: . A text node cannot be a child of a <View>.
```

### 原因

React Native 要求普通文字必须放在 `Text` 组件里。

如果某个表达式最终返回了字符串或空字符串，并且它直接出现在 `View` 下面，就可能触发这个错误。

本项目中遇到的场景是：

```tsx
{loadError && <Text>{loadError}</Text>}
```

当 `loadError` 是空字符串时，React Native Web 可能把它当成文本节点处理。

### 新手解释

可以理解为：

```text
View 只能直接放组件
文字必须先放进 Text
```

空字符串虽然看起来什么都没有，但在某些情况下也可能被当成文本。

### 解决方式

把 `&&` 条件渲染改成明确的三元表达式：

```tsx
{loadError ? <Text>{loadError}</Text> : null}
```

如果没有内容，就返回 `null`。

React Native 可以安全忽略 `null`。

### 以后怎么避免

- 在 `View` 下面渲染文字时，必须使用 `Text` 包裹；
- 对可能是空字符串的条件渲染，优先使用 `条件 ? 组件 : null`；
- 看到 `A text node cannot be a child of a <View>` 时，先检查 `View` 下是否有裸文字、空字符串或错误的条件渲染。

## 8. Expo Web：index.ts.bundle net::ERR_ABORTED

### 现象

浏览器控制台出现：

```text
net::ERR_ABORTED http://localhost:8081/index.ts.bundle...
```

### 原因

这个地址是 Expo Web 自己的前端 bundle，不是后端 API。

常见原因包括：

- Expo 正在重新编译；
- 浏览器刷新时旧请求被取消；
- 开发服务刚重启；
- 缓存中的旧 bundle 被中断。

### 新手解释

可以理解为：

```text
浏览器正在拿前端代码包
但这个请求中途被取消了
```

它和 `GET /api/health`、`GET /api/stocks` 不是一类问题。

### 解决方式

如果页面能正常显示，可以先忽略。

如果页面白屏或不更新：

```text
Ctrl + F5
```

必要时重启前端：

```text
npm run web -- --clear
```

### 以后怎么避免

- 先区分报错地址是前端 bundle 还是后端 API；
- `localhost:8081/index.ts.bundle` 是前端自身问题；
- `127.0.0.1:8000/api/...` 才是前端请求后端问题。

## 9. 找不到 AI 启动的前后端终端

### 现象

AI 启动了前端或后端服务，但用户在 IDE 里找不到对应终端。

### 原因

AI 工具启动的命令可能运行在后台终端中，不一定是用户当前正在看的终端标签。

### 新手解释

可以理解为：

```text
服务确实在运行
但不是你亲手打开的那个终端
```

所以不容易看到日志。

### 解决方式

后续服务类命令推荐用户自己启动。

后端：

```text
cd d:\ai-stock-analysis\ai-stock-app\backend
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000 --reload
```

前端：

```text
cd d:\ai-stock-analysis\ai-stock-app\mobile-app
npm run web -- --clear
```

### 以后怎么避免

- 服务类命令尽量由用户自己启动；
- AI 只提供命令和排错建议；
- 用户保留两个可见终端：一个后端，一个前端。

## 10. PowerShell 不支持 mkdir -p 多目录写法

### 现象

在 Windows PowerShell 里执行：

```text
mkdir -p src/screens src/navigation
```

可能报错。

### 原因

`mkdir -p` 是类 Unix Shell 常见写法，不是 PowerShell 5 的通用写法。

### 新手解释

可以理解为：

```text
不同终端语言不一样
Linux/macOS 常用写法不一定能直接放到 PowerShell 里
```

### 解决方式

PowerShell 中可以分开创建：

```text
mkdir src/screens
mkdir src/navigation
```

或者使用 PowerShell 原生命令。

### 以后怎么避免

- Windows PowerShell 5 中避免使用 `&&`、`mkdir -p` 等类 Unix 写法；
- 执行命令前确认当前 shell 类型；
- 多目录创建可以拆成多个命令。

## 11. 自选股删除按钮点击后没有删除

### 现象

在自选页点击“删除”按钮后，自选股没有从列表中删除。

也可能出现点击删除时页面反而进入任务状态页的情况。

### 原因

本项目中遇到的主要原因是前端交互结构不够稳定：

```text
外层 Pressable：点击股票行创建分析任务
内层 Pressable：点击删除按钮删除股票
```

内外层都是可点击组件时，点击删除按钮可能受到外层点击事件影响。

同时，删除前使用的 `Alert.alert` 在 Expo Web 环境中表现可能不稳定，确认按钮回调不一定按预期执行。

### 新手解释

可以理解为：

```text
一张纸上有一个大按钮
大按钮里面又放了一个小按钮
```

用户点小按钮时，有时大按钮也会以为自己被点了。

### 解决方式

采用最小修复：

1. 删除按钮不再依赖确认弹窗；
2. 点击“删除”后直接调用 `DELETE /api/stocks/{code}`；
3. 长按股票行也直接执行删除；
4. 删除成功后同步清理本地保存的 `stockTaskIds`；
5. 删除成功后同步清理页面上的 `taskStatuses`。

### 以后怎么避免

- 避免在一个 `Pressable` 里面再嵌套另一个 `Pressable` 承担不同业务动作；
- Web 预览阶段如果 `Alert.alert` 表现异常，可以先用直接动作验证接口；
- 删除成功后不仅要刷新列表，也要清理相关本地状态；
- 如果后续要恢复删除确认，建议使用更稳定的自定义确认弹层。
